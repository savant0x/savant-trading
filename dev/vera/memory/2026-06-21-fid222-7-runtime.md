## Design pivot from spec

User spec called for in-loop funnel_inputs collection alongside `pair_data_vec.push`. Three consecutive str_replace attempts at the push-site anchor failed with byte-mismatch errors despite multiple diagnostic dumps. Pivoted to POST-LOOP collection: build funnel_inputs by iterating `active_pairs` post-loop + force-injecting positioned pairs. Architecturally cleaner (no in-loop ordering dependency; all candidate collection at one auditable site; open-position guard works at TWO layers not one). Net wash on correctness; readability improved.

## Risks caught & mitigated

- **Thinker risk #1 (orphaned open position dropped by Top-K)**: dual-layer guard. Pre-filter drops CAN exclude positioned pairs from `pair_data_vec`. Mitigation: post-loop force-inject ALL positions into funnel_inputs (Layer 1) + pair_data_vec.retain(|pd| top_k || positioned) (Layer 2). **Verified safe.**
- **Thinker risk #2 (blocking I/O from sync JSONL write)**: tokio::spawn decouples cycle from disk write. Cycle latency +1ms. **Verified safe.**
- **Thinker risk #4 (stale telemetry on PassThrough)**: record_funnel_runtime REPLACES entire FunnelRuntimeState on PassThrough to defaults + explicit disabled_reason. Dashboard reflects current cycle. **Verified safe.**
- **Score inflation**: force-injected candidates with IndicatorValues::default() score 0.0; if min_score_threshold > 0.0 they get dropped. Retained via Layer 2 but inflates threshold_drop in JSONL. **Carry-forward to FID-222.8 nit #3.**
