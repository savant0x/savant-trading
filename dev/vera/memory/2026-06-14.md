**2. Risk module — well-built, well-tested, partially wired**

`src/risk/` contains 4 files, 44KB total, 30+ inline tests:
- **`circuit_breaker.rs`** — `CircuitBreaker` struct with `check`, `check_with_heat`, `check_full`, `check_per_trade_loss`. 7 tests. **The `check_per_trade_loss` method has zero callers** (confirmed by grep — 1 match, the definition). **Verdict:** The risk module itself is in good shape. The bug is *wiring*: `check_per_trade_loss` is not called from anywhere.

**3. Execution layer — two parallel paths**

`src/execution/` has 3 files: `engine.rs`, `portfolio.rs`, `dex/` (4 files).
- **`engine.rs`** — `ExecutionEngine` trait defining the order interface. Has default no-op implementations for `kill`, `sync_balance`, `place_stop_loss`, `check_liquidity`, `sync_wallet_positions`, `register_position`, `query_token_balance`, `chain_id`. The real money flows through `DexTrader`.
- **`portfolio.rs`** — `PortfolioManager` is the **paper-trading implementation** of the `ExecutionEngine` trait. 912 lines, 9 inline tests. Same self-balance-style contamination risk as the DEX path, but for *price* instead of *balance* — the paper path's PnL is only as honest as the tracked-price feed.
- **`dex/`** — `DexTrader` (the live money path), `ZeroXBackend` (the 0x API client), `InchBackend` (1inch client), `mod.rs` (shared types). The DEX path is where the bugs live.

**Verdict:** The paper path is solid. The DEX path is where the masking bug lives. The two paths are not isolated — they share some types and the engine treats them as interchangeable backends of the same `ExecutionEngine` trait. This is a design strength (testability) but also a design weakness (a bug in one path can be sidestepped by routing to the other).

**4. savant.blocked mechanism — exists, partially wired**

The block file mechanism is in `src/engine/mod.rs` (lines 148, 1515-1531, 3195-3200) and `src/api/mod.rs` (lines 395, 417-435):
- **Startup check** (engine/mod.rs:148-159): Engine refuses to start if `savant.blocked` exists. Reads file contents and errors loudly.
- **Write** (engine/mod.rs:3195-3200): When `circuit_breaker.check()` returns `Triggered`, the engine writes `savant.blocked` with a timestamp, classified `Trigger:` type, and reason. **Supported trigger types** include `daily_loss`, `drawdown`, `max_positions`, `portfolio_heat`, `spread`, `per_trade_loss`, `unknown`.
- **Auto-clear at midnight** (engine/mod.rs:1515-1531): At midnight UTC, if the block file's trigger is `daily_loss`, it's auto-cleared. Other triggers persist.
- **API clear** (api/mod.rs:417-435): `POST /api/risk/clear-block` endpoint deletes the file.
- **Startup auto-clear** (main.rs:311-313): On a fresh start, if the file exists, it's deleted with a log message.

**The `per_trade_loss` trigger_type is in the file format. The `check_per_trade_loss` function exists. The architecture supports it. The wiring is missing on the close path.**

**5. Engine close path — no per-trade loss check**

`src/engine/mod.rs:3260-3370` is the close handler. After `ex.close_position()` returns `Ok(order)`, the engine:
1. Computes `pnl` from `exit_price` and `pos.entry_price`
2. Computes `pnl_pct`
3. Records the trade to journal
4. Updates decision log
5. Updates episodic memory
6. Updates shared state
7. **Does NOT call `circuit_breaker.check_full(pnl, equity)`**
8. **Does NOT call `check_per_trade_loss(pnl, equity)`**
9. **Does NOT write `savant.blocked` even if the loss is catastrophic**

Compare to the entry path at line 3173-3200 which *does* call `circuit_breaker.check(portfolio.account())` and write the block file on trigger. The entry path is wired. The close path is not.

**This is the missing wiring. It is small. It is mechanical. It is the structural fix for LESSON-001's failure mode.**
