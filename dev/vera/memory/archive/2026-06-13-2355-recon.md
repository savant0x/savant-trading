**Verdict:** The risk module itself is in good shape. The bug is *wiring*: `check_per_trade_loss` is not called from anywhere.

### 3. Execution layer — two parallel paths

`src/execution/` has 3 files: `engine.rs`, `portfolio.rs`, `dex/` (4 files).

- **`engine.rs`** — `ExecutionEngine` trait defining the order interface. Has default no-op implementations for `kill`, `sync_balance`, `place_stop_loss`, `check_liquidity`, `sync_wallet_positions`, `register_position`, `query_token_balance`, `chain_id`. **This trait is the abstract interface; the real money flows through `DexTrader` which I read in earlier sessions.**
- **`portfolio.rs`** — `PortfolioManager` is the **paper-trading implementation** of the `ExecutionEngine` trait. 912 lines, 9 inline tests. It has:
  - `place_order` with maker-fee routing (limit at bid/ask when spread > fee differential)
  - `close_position` (full close)
  - `close_position_partial` (TP scale-outs)
  - `check_stops` with trailing stop, TP1 (50% scale + break-even), TP2 (60% of remaining), TP3 (full close), deduplication for repeated closures
  - `dynamic_slippage` based on ATR and book depth
  - `refresh_equity` as single source of truth
  - State save/load to disk
  - **NO on-chain verification.** `on_chain_verified: false` and `tx_hash: None` on every TradeRecord it generates.
  - **Uses `pos.current_price` for close** (line 644), which is set by `update_prices` (line 147). Same self-balance-style contamination risk as the DEX path, but for *price* instead of *balance* — the paper path's PnL is only as honest as the tracked-price feed.
- **`dex/`** — `DexTrader` (the live money path), `ZeroXBackend` (the 0x API client), `InchBackend` (1inch client), `mod.rs` (shared types). The DEX path is where the bugs live.

**Verdict:** The paper path is solid. The DEX path is where the masking bug lives. The two paths are not isolated — they share some types and the engine treats them as interchangeable backends of the same `ExecutionEngine` trait. This is a design strength (testability) but also a design weakness (a bug in one path can be sidestepped by routing to the other).

### 4. savant.blocked mechanism — exists, partially wired

The block file mechanism is in `src/engine/mod.rs` (lines 148, 1515-1531, 3195-3200) and `src/api/mod.rs` (lines 395, 417-435):

- **Startup check** (engine/mod.rs:148-159): Engine refuses to start if `savant.blocked` exists. Reads file contents and errors loudly.
- **Write** (engine/mod.rs:3195-3200): When `circuit_breaker.check()` returns `Triggered`, the engine writes `savant.blocked` with a timestamp, classified `Trigger:` type, and reason. **Supported trigger types** include `daily_loss`, `drawdown`, `max_positions`, `portfolio_heat`, `spread`, `per_trade_loss`, `unknown`.
- **Auto-clear at midnight** (engine/mod.rs:1515-1531): At midnight UTC, if the block file's trigger is `daily_loss`, it's auto-cleared (PnL resets). Other triggers persist.
- **API clear** (api/mod.rs:417-435): `POST /api/risk/clear-block` endpoint deletes the file.
- **Startup auto-clear** (main.rs:311-313): On a fresh start, if the file exists, it's deleted with a log message.

**The `per_trade_loss` trigger_type is in the file format. The `check_per_trade_loss` function exists. The architecture supports it. The wiring is missing on the close path.**

### 5. Engine close path — no per-trade loss check

`src/engine/mod.rs:3260-3370` is the close handler. After `ex.close_position()` returns `Ok(order)`, the engine:

1. Computes `pnl` from `exit_price` and `pos.entry_price`
2. Computes `pnl_pct`
3. Records the trade to journal
4. Updates decision log
5. Updates episodic memory
6. Updates shared state
7. **Does NOT call `circuit_breaker.check_full(pnl, equity)`**
8. **Does NOT call `check_per_trade_loss(pnl, equity)`**
9. **Does NOT write `savant.blocked` even if the loss is catastrophic**

Compare to the entry path at line 3173-3200 which *does* call `circuit_breaker.check(portfolio.account())` and write the block file on trigger. The entry path is wired. The close path is not.

**This is the missing wiring. It is small. It is mechanical. It is the structural fix for LESSON-001's failure mode.**

### 6. savant.blocked file on disk (actual current state)

`savant.blocked` last written 2026-06-13 12:20:05 UTC, contents:
```
2026-06-13T16:20:05.968393700+00:00
Trigger: max_positions
Reason: Max positions reached: 1
```

Note: `2026-06-13T16:20:05` is UTC. 12:20:05 PM EST = 16:20:05 UTC. So this was the engine writing the block when it had 1 open position (the GRT long) and tried to enter another. The engine refused the entry. The file remained. The next startup sees the file, errors out (engine/mod.rs:148-159), and refuses to run.

Wait — there's something I need to flag. The startup check at line 148-159 looks like it should have *prevented* the engine from running on the morning of the incident. But the incident happened earlier (12:20:05 PM is the *end* of the morning, not the start). The file existed *during* the drain. The engine was running. The file didn't block it. This is because the file is checked *only at startup*. Once the engine is running, writing the file doesn't stop the next cycle. The file is a *restart gate*, not a *runtime halt*.

This means: **if the engine ever wrote `savant.blocked` mid-session for any reason, it would still keep trading until the next restart.** This is a design gap. The block file should also be checked at the *start of every cycle*, not just at startup. Or the engine should poll the file. Or the breaker should set an in-memory flag that is checked every cycle.

| RAIN/USD | Short | 12.73 | -$0.00050 | -0.3% | $0.00050 |

The `pnl_pct: -0.3%` matches `fee_rate = 0.003` (0.3% per side, or 0.3% round-trip — depends on fee model). The `exit_price == entry_price`. The "loss" is *only the trading fee*. The position was closed at the entry price, the verifier was faked to look like entry price, and the engine recorded "I lost the fee."

The 4 closed trades represent 4 closing transactions. The "24 swaps" in the incident report are the *attempted* transactions — buys + closes + retries. The 4 closed trades are a subset. The full loss picture includes the **outgoing USDC that didn't come back** ($351.54 out, $366.52 in, but $0.02 net is wrong — the wallet says $0, not -$0.02; the "balance" field in dex_state says `0.051443` which is the GRT remaining). The real loss was the unrecovered portion of the 24 swaps.

**The dex_state.json is more honest than the comment in the code.** The comment says "breakeven." The PnL field says "-0.3% fee." The fee IS recorded. The position is removed. But the *aggregate* loss is hidden by the engine's lack of on-chain balance reconciliation (Nova's N4 finding — `usdc_balance_before = self.balance`, contamination compounds).

### 8. Phantom position

`dex_state.json:3-19` shows one open position: `dex-38-GRT_USD`, side Long, 639.54 GRT, entry_price 0.01987, opened 2026-06-13 09:27:47. The wallet has ~5.9 GRT on-chain. The phantom is 639.54 vs 5.9. **The position is 108x the actual holdings.**

Stop loss: 0.0. Take profits: 0.0. Risk amount: 0.0. Strategy: `dex_0x`. The position has *no risk parameters* — the engine entered it without setting a stop or TP. **This is the open risk.** If GRT price moves 1%, the engine has no stop to cut the loss. The position will ride to whatever happens.

The position is also entry 0.01987 and current_price 0.01987 (no price update). The bot treats this as "still at entry." If the price has moved at all since 09:27:47 UTC, the engine doesn't know.

**The phantom is real and dangerous.** The wallet has 5.9 GRT, the bot thinks it has 639.54. If the bot tries to *close* this position, it will fail because the wallet doesn't have the tokens to sell. If the bot tries to *add* to this position, it will fail for the same reason. If the bot does nothing, the position sits forever, immune to the loss because there's no on-chain reconciliation.
