## What Nova got right (and I missed)

### N1: verify_swap_direction is binary, not magnitude — CONFIRMED

I read `src/execution/dex/trader.rs:264-351` (88 lines of the function). It is boolean logic:
- `src_sent_from_wallet: bool` — did src leave the wallet? yes/no
- `dst_received_by_wallet: bool` — did dst arrive at the wallet? yes/no

It does not check whether the *amount* of dst matches the 0x quote. For the GRT siphon (58.7 GRT siphoned to `0xf5c4F3Dc...`, 3.3 GRT residual to wallet), this function sees 3.3 GRT arrived → returns `Ok(())`. The siphon is invisible to it.

Nova is also right that the post-mortem implies this would have caught the siphon. It would not have. The post-mortem was wrong on this point. Nova corrected it.

**Implication:** FID-105 is necessary but not sufficient. A magnitude check (Layer 2) is the only thing that catches a partial-swap siphon. The 0x quote said "you'll get X tokens" — we need to verify we got ~X tokens, not "≥0 tokens."

### N2: drain_retry_queue is dead code — CONFIRMED

I grepped `drain_retry_queue` — 1 match. The definition at `trader.rs:748`. Zero callers.

I also grepped `retry_queue` — 7 matches. The queue is *pushed to* at line 729. It's *initialized* at line 487 and 569. But the function that *drains* it (`drain_retry_queue` at 748) is never called.

What this means: when a close swap fails (dust, RPC error, transient), the retry queue accumulates. Nothing pulls from it. The only way a position gets removed is through the FID-146 breakeven-masking path. If we fix the masking (per Nova's Layer 2 proposal), we have a new problem: positions that fail to close have no recovery path. They'll sit in the in-memory map forever, except the next session-load will resurrect the on-chain state.

**Implication:** N2 and the FID-146 fix are coupled. We can't just halt on close failure — we need a recovery path for halted-position close failures. Otherwise the fix creates a new deadlock (position open, can't close, engine halted, manual recovery needed for every single position).

### N3: FID-146 has two distinct meanings — CONFIRMED

**Recommendation (proposed, not adopted):** The jury veto feature should be renamed to a new FID number. The trader.rs masking fix should keep the FID-146 number for continuity (since FID-146 is the document that introduced the masking).

### N4: usdc_balance_before = self.balance — CONFIRMED, AND IT COMPOUNDS

I read `trader.rs:1737` directly: `let usdc_balance_before = self.balance;`. Nova is correct that this is in-memory, not on-chain.

The compounding part: every FID-146 masked trade updates `self.balance` via `self.balance = usdc_balance_before + verified_proceeds;` at `trader.rs:1918`. Since `verified_proceeds` is 0 (masked), `self.balance` is set to `usdc_balance_before` (the contaminated in-memory value). After 24 swaps with masked losses, `self.balance` could be wildly divergent from on-chain reality.

This means **Nova's Layer 2 proposal needs a prerequisite fix.** Her pseudocode:
```rust
if !verified || gained < sent_usd * 0.90 { halt }
```
needs `gained` to be measured against a *clean* baseline. The clean baseline is the on-chain `balanceOf` query. Without that, the 90% check is comparing a real RPC result against a contaminated in-memory number, and the contamination gets worse every masked loss.

**Implication:** The fix order matters:
1. Fix N4 first (baseline = on-chain, not self.balance)
2. Then fix the masking (Layer 2 halt-on-failure)
3. Then add Layer 1 (independent oracle) as a separate defense
4. Calldata inspection as defense in depth
5. drain_retry_queue wiring as recovery (otherwise we have a deadlock)

## What I verified that Nova didn't add but is consistent with her findings

Looking at her `verified_proceeds` flow more carefully (I had read this earlier), there's a third way `verified_proceeds` can become 0.0 — at line 1832-1844:
```rust
if gained <= 0.0 {
    // Dust return — close succeeded on-chain but delivered $0.
    error!("FID-146: Close tx {} delivered ${:.2} USDC (dust return). \
            Trusting on-chain close, removing position with breakeven PnL. \
            Tokens may be stranded — check wallet.",
            tx_hash, gained);
    proceeds = 0.0;
    verified = false;
    break;
}
```

The comment says "Tokens may be stranded — check wallet." This is the engine *knowing* the position may be stranded and *deciding* to record it as breakeven anyway. This is worse than I had it in my prior analysis. The engine has *two* masking paths: verification failure and dust return. They produce the same output (proceeds=0, exit_price=entry, breakeven PnL), but they enter through different conditions.

The 4 trades in the post-mortem all hit one of these two paths. None of them recorded a real loss. None of them wrote `savant.blocked`.
