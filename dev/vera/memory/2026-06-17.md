## What happened, in order

### ~00:45-01:15 EST — FID-175: start.bat kills kilo CLI

Spencer ran `start.bat` and his kilo CLI session froze. The `Stop-Process -Name node -Force` (line 36) was killing kilo's MCP server processes. Fix: scoped `node.exe` kill to only processes with `savant` in their command line.

**Lesson:** `Stop-Process -Name node` kills ALL node processes, not just the app's. Scope by command-line filter.

### ~01:15-01:35 EST — FID-176: dotenvy parse failure on 0X_API_KEY

Spencer's `.env` had `0X_API_KEY=611d1892...` (starts with digit). The dotenvy parser rejected it as an invalid line, which caused the entire `.env` file to fail to load → all API keys empty → 401 errors on every LLM call. Fix: commented out the `0X_API_KEY` line (not needed — we use ZEROEX_API_KEY) and documented the gotcha in `.env.example`.

**Root cause:** The `0X_API_KEY` line was from an earlier API key format. It started with a digit (`0X_`), which the dotenvy parser treats as invalid, and dotenvy's default parser is strict: a single bad line poisons the whole file.

**Lesson:** Stray environment variable lines with leading digits can silently break ALL env loading. If you see 401s on every service, check `.env` for parse failures.

### ~01:35-01:45 EST — FID-178: start.bat Anvil block cmd.exe parse error

The Anvil auto-start condition (`%SAVANT_CONFIG:anvil=%` string check) had Windows cmd.exe nested-if parsing issues. Replaced the whole block with `call start-anvil.bat` unconditionally — start-anvil.bat is already idempotent (detects Anvil at port 8545 before launching).

## Key lessons from this session

- **"Nothing is out of scope" — scope expands to include bugs found along the way.** FID-181 started as "equity curve fix" and grew to include dashboard layout, warning cleanup, and WebSocket v2 because those were also broken. Spencer's rule: never defer unless explicitly stated.
- **4 issues can be 1 FID when they're all found in a single session and none is a blocker for shipping.** FID-181 is 4 unrelated fixes bundled into one atomic commit. The Perfection Loop validates each independently, and all share the same release.
- **The equity curve was broken from inception.** The engine never wrote to `state.shared.equity_curve`. Only backtest did. For ~3 releases the dashboard was permanently "Collecting equity data…" with no one noticing because the engine was always off during Vera's sessions. **Lesson: test the monitoring infrastructure even when the engine is in paper mode.**
- **Atomic write matters for data integrity.** `/tmp/equity_history.tmp` + `std::fs::rename` = either old file or new file, never a half-written file. The 200-snapshot cap prevents unbounded file growth.
- **WebSocket protocol errors hide behind null fields.** Kraken v2 returns `result: { channel: null }` on error. The response handler was reading `result.channel` instead of checking `error`. **Lesson: WS handlers should check `error` field first, not `result`.**
- **HashSet dedup for per-cycle log lines.** The GoPlus warning "no known address" fired once per token per cycle. With 21 tokens, that's 21 warn lines/cycle. A `HashSet<String>` with `logged_unknowns` deduplicates: log once per token ever. **Pattern: cached log suppression with per-component sets.**
- **The engine was running for 35+ minutes before I realized the equity curve was still empty.** The fix required a new build + restart. Paper mode makes this cheap, but in production every restart loses 5 min of data. **Lesson: verify the monitoring telemetry in the first cycle, not after an hour of data.**
- **Dashboard visual issues are often stale server processes.** The new build (with grid-cols-3 and row-span-3) is in `.next/BUILD_ID cAbJiH_K7vHbqUBzD6lOX` but the dashboard server may have been started before that build. `npm run dev` auto-rebuilds; the server process needs restart to pick up new pages. **Lesson: after dashboard changes, kill the next-server process before restarting start.bat.**
- **Jury re-enablement was trivial (config change) but conceptually important.** FID-179 just toggles `jury.enabled = true` in test-anvil.toml. But the jury system was disabled because of my earlier misconception (FID-161 era). Spencer: "the whole logic of the jury was getting multiple opinions, that's why i built the system lol." The jury is the differentiator. **Lesson: don't disable core architecture features based on LLM reasoning noise issues — fix the noise instead.**
- **Hyperliquid is parked, not dead.** `dev/vera/notes/hyperliquid-research-2026-06-17.md` documents the KY/US legal question. `dev/vera/specs/v0.15.0-scope-parallel-multi-chain.md` outlines path to v0.15.0 with HL after parallel multi-chain. Per Spencer: "organize it outside of FIDs." **Pattern: `dev/vera/notes/` for research, `dev/vera/specs/` for plans, never FIDs for parked work.**
