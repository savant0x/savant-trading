## The 9 blindspots I found

### FID-168 v2 (5 improvements)
- **A. Snapshot data gap.** The cycle_snapshot only captured `pair | action | conf`. The summary prompt (FID-165) asks for "current regime, ATR, RSI, vol, memory highlights" — but the data wasn't captured. **The summary was operating on partial data.** v2 captures all the fields.
- **B. Cycle watchdog risk.** The summary call adds 5-60s. The 5-min cycle watchdog at line 5198 could trip if the cycle is already at 4min. **No safety check.** v2 adds `if elapsed > 240s: skip` with a warning log.
- **C. Dead code.** `SummaryContext::is_stale()` existed but was never called. **Dead code is a smell.** v2 calls it. Force-resummarize every 60s.
- **D. Math wrong in FID.** v1 estimated first pruning at ~100 cycles. v2: at 30 pairs × 70 chars = 525 tokens/cycle, target=5000, first pruning at ~10 cycles. **Off by 10x.** Updated.
- **F. Wrong mental model.** v1 said we summarize the pruned (removed) blocks. **Actually we summarize the REMAINING blocks (recently-active).** The LLM wants the live context, not the historical archive. Updated FID.

### FID-170 v2 (2 improvements)
- **A. Doesn't use `summarize_with_fallback` per stage.** v1 used `self.summarize(stage)` per stage, losing the partial-failure recovery. **Real fidelity gap vs openclaw.** v2 uses `self.summarize_with_fallback_public(chunks)`.
- **B. Count-based splits.** v1's `split_into_stages` divided by count, leading to lopsided stages. v2 added `split_into_stages_by_tokens` using greedy fill. **Balanced LLM inputs regardless of input distribution.**

### FID-171 v2 (2 improvements)
- **A. Dead code removed.** v1 had `let _ = chunk_size_cap;` which computed and immediately discarded its value. **Two cycles later, the placeholder was still there.** v2 removes the dead line and uses the chunked path properly.
- **C. "You are the new LLM" role statement.** v1's HANDOFF_INSTRUCTIONS talked about "the new model" in third person. **v2 addresses the LLM as "you" and includes a "YOUR ROLE" section.** A new LLM now knows its role.

## Lessons from this session

- **The Perfection Loop on existing FIDs is as important as on new ones.** Spencer's question "if they don't have FIDs" was a hook for "do the strict-read pass." The 9 blindspots I found are quality issues that don't break tests but reduce the value of the feature. **A 5-minute strict-read pass after the initial implementation is high-leverage work.**
- **Most "out of scope" improvements turn out to be in scope.** The 9 blindspots include: dead code, wrong math, wrong mental model, missing data, missing safety check. **None of these were "future enhancements" — they were concrete bugs in the shipped code.** Spencer's "include any missed suggestions or blindspots and improvements i forgot to ask" was the trigger.
- **Strict-read is its own kind of work.** It's not "make it work" or "make it fast" — it's "find what the v1 author got wrong." Required mental mode: "this is finished, now look for what's wrong with it." Different from the greenfield mindset.
- **The "snapshot data should match summary prompt fields" lesson is fundamental.** v1 captured `pair | action | conf`. The summary prompt asked for regime/ATR/RSI. **The summary was operating on partial data.** Lesson: **the data flow into a summarization step must match the prompt's expectations, or the summary is degraded.** This is a general principle, not just for this FID.
- **Auxiliary LLM calls need a watchdog safety.** Always check `cycle_start.elapsed()` before invoking a slow operation. Pattern: if elapsed > 4min, skip and log. The data flows back next cycle.
- **"Use or remove it" — dead code is a smell.** v1 had `is_stale()` defined but never called. v1 had `let _ = chunk_size_cap;` which discarded its value. **Two cycles later, the placeholder was still there.** Lesson: **if you add a method or field "for later," use it in the same FID or remove it. Otherwise the placeholder accumulates.**
- **"Simpler than openclaw" is not always better.** v1 used `self.summarize(stage)` instead of `self.summarize_with_fallback(chunks)`. **The "simpler" version had weaker failure recovery.** A stage with oversized blocks would have failed in v1; v2 retries with the non-oversized subset.
- **Token-based splits vs count-based splits matter for LLM-balanced inputs.** 100 blocks × 1 part = 1 stage of 100 blocks. If 50 of those blocks are 5 tokens each and 50 are 100 tokens each, the LLM sees 250+5000 = 5250 tokens in one call. **Token-based splits keep each stage under target_per_stage tokens.** The LLM gets a balanced input regardless of input distribution.
- **Address the LLM directly in the prompt.** v1's instructions said "the new model" in third person. v2 says "You are the new LLM." **The LLM is reading the prompt; address it as "you."** Pattern: prompts that roleplay a specific actor should use second-person.
- **Public API + private helper pattern.** v2 added `summarize_chunks_only` as a private helper that takes `&LlmProvider`, mirroring the existing private `summarize_chunks`. The public `summarize_for_handoff` orchestrates: chunk → `summarize_chunks_only` → return. **Pattern: keep helpers private, expose one public method that orchestrates them.**
- **Math claims need verification.** v1 said "first pruning at ~100 cycles." v2: at 70 chars/pair × 30 pairs × 5 cycles = 10500 chars/cycle, target=5000 → first pruning at ~10 cycles. **The v1 estimate was off by 10x.** Always run the math before claiming a behavior.
