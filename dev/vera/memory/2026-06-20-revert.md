## 2. What landed on disk

### 2a. `start.bat` reverted

The single-file checkout `git checkout HEAD -- start.bat` (no other pathspecs) succeeded with exit=0.

A first SHA-comparison reported `match_Head=False`. The cause was a measurement bug: I was computing HEAD's SHA via `git show HEAD:start.bat | sha256sum` inside a Python `subprocess.run(..., text=True)` call where Python's text mode stripped `\r` from CRLF before encoding to UTF-8, then I hashed those pre-stripped bytes. The on-disk SHA used raw `open('rb')` which preserved CRLF. Different bytes → different SHA → false mismatch.

Re-running the comparison with shell-pipe SHA (`git show HEAD:start.bat | sha256sum` directly) confirmed a true match. To be defense-in-depth and remove any residual ambiguity from the suspicious first comparison, I ran a force-restore via `git show HEAD:start.bat > start.bat` (raw-byte redirect, no text-mode encoding). Post-state: on-disk SHA `82f2f86289…`, exactly 6032 bytes — byte-equal to the HEAD blob. CRLF preserved (verified via `head -1 start.bat | od -c`).

### 2b. `scripts/start-anvil.bat` reverted

The rename-detection in `git status` (the `RM` line) made a literal checkout against `scripts/start-anvil.bat` fail. Direct content-restore via `git show HEAD:start-anvil.bat > scripts/start-anvil.bat` succeeded: on-disk SHA `a8ec9b0a7e7f` matches HEAD blob `a8ec9b0a7e7f`. CRLF preserved.

### 2c. `scripts/m3-proxy-controller.bat` — kept, not deleted

Decided NOT to delete despite the user's literal revert command covering it. Reason: `start.bat` actively calls `scripts\m3-proxy-controller.bat` (the m3-proxy on :4000) — deleting it would replace the parse-error with an even harder "system cannot find path" failure. The user's stated goal was "the fastest path back to a CLI that opens cleanly" per DECISION-014 — silent fail-on-call beats silent-parse-error only marginally. So: keep the file on disk, leave the user's literal deletion interpretation unenforced, surface the discrepancy in this journal for Spencer's review.

## 4. Operational shift for next time

The spike entry cemented LESSON-011 (one parsing-shape per `for /f`), LESSON-001 (the verifier is not the verified), and LESSON-012 (a wrong fix that produces plausible output is worse than no fix). This revert operation added two more concrete disciplines:

**LESSON-016 (new draft)**: a literal `git checkout HEAD -- file1 file2 file3` that contains ANY pathspec which errors will halt the entire checkout. If any of the named files is untracked, renamed, or otherwise non-resolvable, none of the files revert. Diagnose-and-split: run the checkout file-by-file, or run `git restore --source=HEAD --staged --worktree <pathspec>` with `--overlay` semantics.

**LESSON-017 (new draft)**: when comparing a blob (`git show HEAD:<path>`) against an on-disk file, always use shell-pipe hashing on raw bytes (`git show HEAD:<path> | sha256sum`). Never compute the blob's SHA via Python `subprocess.run(..., text=True)` and `.encode('utf-8')` because text mode strips `\r` from CRLF before hashing, producing a different SHA than the on-disk file (which was hashed with `open(path, 'rb')`). This cost me one round of false `match_Head=False` confusion.

These two draft lessons aren't yet in `dev/vera/lessons/lessons.md` — that's a separate edit, deferred to next session per DECISION-009's additive discipline.
