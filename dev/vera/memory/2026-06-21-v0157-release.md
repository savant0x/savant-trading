## Post-Ship Hotfix — FID-225 Reconciliation Per-Token Threshold Decouple

**Status**: HOTFIX APPLIED (working tree, uncommitted) | not yet pushed | validated locally
**Operator**: Spencer (after v0.15.7 archive)
**Subagent**: Buffy (Codebuff CLI substrate, minimax-m3)
**Files modified (uncommitted)**:
- `src/execution/reconciliation.rs` — added `token_divergence_threshold_usd: f64` field with `#[serde(default)]` to `ReconciliationConfig`; added entry in `Default` impl (= 5.00); switched line ~215 per-token divergence check from `divergence_threshold_usd` → `token_divergence_threshold_usd`; added regression test `token_threshold_is_decoupled_from_usdc_threshold`
- `src/engine/mod.rs` — two construction sites updated to include `token_divergence_threshold_usd: 5.00`
- `tests/fid212_close_reconciliation.rs` — `default_cfg()` helper updated

### Trigger

Engine halted on cycle 34 of 2026-06-21 3:35 AM run with `WALLET_RECONCILIATION_HALT [real-time]` log:

```
[ERROR] [Reconciliation] WALLET_RECONCILIATION_HALT [real-time]: Wallet reconciliation divergence (RealTime):
in_memory_usdc=$50.0000, on_chain_usdc=$50.0000 ($0.0000 / 0.00% divergence,
thresholds: $0.1000 / 1.00%). 1 token(s) with divergence.
```

USDC balances matched exactly ($50=$50). Halt was triggered by per-token divergence — the leading log line (`34, on-chain=0.000000, div=$5.2500`) shows a held token with $5.25 on-chain divergence. Heartbeat also reported key-manager cleanup of 10 jury keys before halting.

### Root cause

`src/execution/reconciliation.rs` had a single `divergence_threshold_usd` (default `$0.10`, intended for USDC reconciliation) being used for BOTH:
- USDC divergence check (correct: $50 in-memory vs $50 on-chain, must match within $0.10)
- Per-token divergence check (incorrect — Anvil price-feed noise routinely produces $0.10–$0.50 USD divergence on small positions)

Result: any token with > $0.10 divergence silently halts the engine, even though USDC is fine. The runtime bug is architectural — same threshold was used for two semantically distinct checks.

### Fix

Decoupled the thresholds. New `ReconciliationConfig` exposes:

```rust
pub divergence_threshold_usd: f64,      // unchanged — $0.10 — USDC only
pub token_divergence_threshold_usd: f64, // NEW — $5.00 — per-token only
```

`#[serde(default)]` on the new field maintains TOML backward compat (existing `config/{default,canary,test-anvil}.toml` `[reconciliation]` blocks still deserialize — new field falls back to Default::default() = $5.00). All 3 explicit construction sites updated to include `token_divergence_threshold_usd: 5.00`.

## Session 2026-06-21 03:50 — FID-225 Round 2: Phantom-Position Classification

**Status:** SHIPPED (working tree, uncommitted) | validated locally | fixes the bot-still-crashing runtime bug from round 1
**Subagent:** Buffy (Codebuff CLI substrate, minimax-m3)
**Files modified:** `src/execution/reconciliation.rs` (helper extraction + inline refactor + 5 regression tests)

### Trigger

After the round-1 hotfix shipped (token_divergence_threshold_usd: $0.10 → $5.00), Spencer's next Anvil run STILL crashed on cycle ~34 with the SAME halt pattern. New log:

```
[WARN] [Reconciliation] WALLET_RECONCILIATION: ai-1 divergence — in-memory=28.332434, on-chain=0.000000, div=$5.2500
[ERROR] [Reconciliation] WALLET_RECONCILIATION_HALT [real-time]: ... (10 tokens(s) with divergence)
```

The divergence WAS just above the new threshold ($5.25 > $5.00). Closer inspection revealed this is **NOT a price-feed noise issue** — it is a **phantom-position**: engine believes it holds 28.33 units of `ai-1`, but the actual Anvil wallet has 0.

### Root cause

`reconcile_wallet_state` per-token check treated ALL divergences homogeneously. The fact that the wallet genuinely has zero of a token the engine thinks it owns (stale-state, reverted swap, prior-session crash residue) is a different class of bug than on-chain drift on a real position. The threshold bump in round 1 was insufficient because every $50-account Anvil session accumulates phantoms.

### Architectural insight

`src/execution/wallet_recovery.rs` (`ChainPositionRecovery`) + `src/execution/reconciliation.rs` (`apply_to_portfolio`) ALREADY have the correct phantom-handling path: a 5-min cycle that purges in-memory-but-not-on-chain positions, with a `safety_halt_threshold_pct = 50%` upper bound (phantoms dominating portfolio = real bug). The heartbeat just needed to NOT CLASSIFY phantoms as haltable drift in the first place.

### Fix (one file, three sections)

1. **Pure helper `is_haltable_token_divergence(expected, on_chain, div_usd, threshold)`** — extracted from inline logic for testability. Returns `true` only for real drift; `false` for phantoms OR sub-threshold.

2. **Heartbeat refactor** — `Ok(on_chain_qty)` arm now calls the helper. Real drift → existing `divergence` log + push to `tokens_with_divergence` (halts as before). Phantom → new `PHANTOM_POSITION` log + skip push. `safety_halt_threshold_pct` at the 5-min recovery path remains the upper bound for genuine bug detection.

3. **5 regression tests** in `mod tests`:
   - `phantom_position_is_not_haltable_divergence` — the exact cycle-34 input (28.33 / 0.0 / $5.24 / $5.00 threshold → false)
   - `real_drift_above_threshold_is_haltable` — 100 vs 85 tokens @ $1.0 = $15 → true
   - `below_threshold_is_not_haltable` — 100 vs 99.5 = $0.5 < $5.00 → false
   - `both_zero_is_not_phantom_and_not_haltable` — edge: empty-state no-op
   - `near_zero_on_chain_with_large_expected_halts_as_drift_not_phantom` — dust residue edge (0.001) classifies as real drift, not phantom

4. **DRY nit** (after first code-reviewer round) — bind `let is_phantom = on_chain_qty == 0.0 && expected_qty > 0.0;` once, reuse in `else if is_phantom`.

### Lessons

- **Symptom-fix detach is the right reflex when root cause is architectural.** Round-1 threshold bump ($0.10 → $5.00) is mechanically defensible but architecturally wrong — it conflates price-feed noise (real drift) with stale-state artifacts (phantom positions). When the first fix doesn't fully resolve, the right move is to re-diagnose the divergence CLASS, not bump the threshold further.
- **Existing infrastructure often has the right answer already; the bug is in the missing wire.** `apply_to_portfolio` (FID-196) handles phantom positions correctly at the 5-min cycle. The heartbeat just needed to STOP premature halts on phantoms so the recovery path could do its job. Lesson: when a feature exists but isn't wired, the fix is to STOP the wrong-path early termination, not to duplicate the feature.
- **Pure helpers + exhaustive test matrices catch regression cheaply.** 5 test cases × pure helper = full branch coverage with zero RPC fixtures. The cycle-34 input was pinned at false so a future regression that re-introduces phantom halts fails loud.
- **DRY isn't optional in Rust.** The first-pass code reviewer correctly flagged the inline `else if on_chain_qty == 0.0 && expected_qty > 0.0` as a duplicate of the helper's internal check. Fixing it post-review is one line; leaving it would have made the helper change risk-prone. **Rule:** extract to helper AND refactor call sites in the same edit, not separate edits.
- **Defense layers stack correctly when each has its own scope.** Round 1 raised the threshold (real-drift noise filter). Round 2 added phantom classification (state-class filter). Round 3 candidate (operator-routed): raise Anvil-mode threshold. Each handles a distinct scenario; none overlaps.
