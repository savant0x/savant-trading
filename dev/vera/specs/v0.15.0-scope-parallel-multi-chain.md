## TL;DR

v0.15.0's headline feature is **parallel multi-chain operation**. Today the engine is single-chain-at-a-time — it picks one chain per cycle (via `SAVANT_CHAIN` env var) and runs all pair evaluation on that chain. v0.15.0 changes this: the engine fans out to all enabled chains in parallel, each with its own sub-cycle, then aggregates results. This unlocks: liquid-major coverage (Ethereum, Base, Optimism, Arbitrum, BSC simultaneously), parallel liquidation-cascade strategies, and the foundation for adding Hyperliquid as a venue in v0.15.0+ or v0.16.0.

Spencer: "we need to properly outline the 15 scope and set aim to expanding full chains so we're not single chain locked. That alone will give us more trade possibilities."

---

## Why this is the v0.15.0 unlock (not v0.14.x)

- **The engine has been single-chain since inception.** v0.13 added multi-chain config (`[chains.*]` in TOML). v0.14 added chain-aware execution (per-chain RPC). But the engine loop itself still picks one chain per cycle.
- **v0.14.2 (FID-167) made the chain choice configurable** via `SAVANT_CHAIN` env var. Default was ethereum (later reverted to arbitrum in FID-177 for Anvil workflow).
- **The actual engine loop** in `src/engine/mod.rs:run()` is the bottleneck. It iterates over `active_pairs` ONCE, on the chosen chain. To go parallel, it needs to iterate ONCE per enabled chain, with `tokio::spawn` per chain.
- **FID-169 was the placeholder for this work.** It was deferred in the 2026-06-16 session because the scope was too large. v0.15.0 is when we tackle it.

## What's in scope for v0.15.0

### Core: parallel multi-chain engine loop

The engine's `run()` loop currently does:

```rust
loop {
    let active_chain = config.chains.get(&SAVANT_CHAIN);
    // ... load pairs for active_chain
    // ... evaluate pairs serially
    // ... execute trades serially
    sleep(interval).await;
}
```

**v0.15.0 changes this to:**

```rust
loop {
    let enabled_chains: Vec<&ChainConfig> = config.chains.values().filter(|c| c.enabled).collect();
    let sub_cycles = enabled_chains.iter().map(|chain| {
        tokio::spawn(async move {
            // ... load pairs for THIS chain
            // ... evaluate pairs serially (within this chain)
            // ... execute trades serially (within this chain)
        })
    });
    for result in futures::join_all(sub_cycles).await {
        // ... aggregate results
    }
    sleep(interval).await;
}
```

**Each chain runs in its own tokio task. Per-pair evaluation within a chain stays serial (to avoid nonce conflicts and chain-specific race conditions).** Cross-chain is fully parallel.

### Per-chain state isolation

Today the engine shares one `EngineState` across the cycle. With parallel chains, each sub-cycle needs isolated state:

- **Per-chain balances** (paper mode: derived from per-chain config; live mode: derived from per-chain RPC)
- **Per-chain positions** (already in `executor_position_map` keyed by symbol; needs to be keyed by chain+symbol)
- **Per-chain executor** (already a HashMap; needs to be populated per sub-cycle)
- **Shared global state** (cycle counter, total P&L, dashboard state) stays in the parent task

**The `src/core/shared.rs` SharedEngineData struct needs to grow per-chain sub-fields**, or become a `HashMap<ChainId, EngineState>`. This is the bulk of the refactor.

### Cycle budget

Today: 5-min cycles. v0.15.0 stays at 5-min cycles, but the engine now runs ~5 sub-cycles in parallel. Each sub-cycle can take up to 5 min before the cycle watchdog at `engine/mod.rs:5243` skips summary.

**Total per-cycle latency: max(per-chain cycle time)** — not sum. So if Ethereum takes 60s and Arbitrum takes 90s, total cycle is 90s, not 150s. The 5-min budget still holds.

### Cost

Per-chain cycle = 1 LLM call (batch) + 1 RPC call per pair + 1 0x quote per pair. With 5 chains, that's 5x LLM calls + 5x RPC + 5x 0x quotes. **Per-cycle cost grows ~5x.** M3 is free. 0x is $0.01/quote → 5 chains × 18 pairs × 12 cycles/hour = $10.80/hour at full load. **Spencer has $0 currently, so this is paper-mode cost only. Real money concerns come later.**

### Config schema changes

```toml
[chains]
# v0.15.0: all enabled chains run in parallel per cycle
active_chains = ["arbitrum", "ethereum", "base", "optimism", "bsc"]

[chains.arbitrum]
chain_id = 42161
enabled = true  # was already there
# ...

[chains.ethereum]
chain_id = 1
enabled = true  # flip from false to true
# ...
```

**The `SAVANT_CHAIN` env var becomes a single-chain override**, not the default. If set, only that chain runs. If unset, all enabled chains run in parallel.

### Dashboard changes

The dashboard's `[ai.openrouter]` config and cycle display need per-chain breakdowns:
- Per-chain balance
- Per-chain position count
- Per-chain decision count
- Per-chain latency

The current single-chain dashboard at `dashboard/src/app/page.tsx:381` already uses a 3-column bento grid. The per-chain section becomes a 4th row or expands the existing rows.

## What's NOT in scope for v0.15.0

- **Hyperliquid integration** — separate workstream (FID-HL-1 through FID-HL-5 per the Hyperliquid research note). HL needs parallel multi-chain as a prerequisite.
- **Cross-chain arbitrage** — the engine evaluates each chain independently. Cross-chain arb (e.g., ETH cheaper on Arbitrum than on Ethereum) is a different strategy. Out of scope.
- **Cross-chain portfolio aggregation** — each chain has its own paper balance. The dashboard shows "Total: $X across N chains" but the engine doesn't pool balances. Out of scope.
- **Per-chain strategy overrides** — the strategy is global. Different pairs on different chains can have different strategy profiles (already supported via `[trading.pairs]`), but the strategy itself doesn't fork per chain. Out of scope.
- **HyperBFT integration** — only relevant if we add Hyperliquid. Out of scope.
