## Summary

## Detailed Description

### Problem

The `ExecutionEngine` trait defines 4 methods: `place_order()`, `close_position()`, `open_positions()`, `balance()`. Only `PaperTrader` implements this. There is no real Kraken execution path. The engine cannot place actual orders.

### Expected Behavior

A `KrakenTrader` that:
1. Places real limit/market orders on Kraken via REST API
2. Tracks fills and updates positions
3. Manages stop-losses (standalone Kraken stop orders, not client-side)
4. Handles partial fills, order rejection, and API errors
5. Polls for order status updates
6. Syncs balance from Kraken account

### Root Cause

No Kraken private API client exists. The current `KrakenClient` only uses public endpoints (OHLC, order book, asset pairs). Private endpoints (AddOrder, CancelOrder, Balance, OpenOrders, ClosedOrders) require HMAC-SHA512签名 and are not implemented.

### Evidence

```rust
// src/execution/engine.rs — only PaperTrader implements this
pub trait ExecutionEngine: Send + Sync {
    async fn place_order(&mut self, pair: &str, side: Side, quantity: f64, price: Option<f64>) -> Result<Order, ExecutionError>;
    async fn close_position(&mut self, position_id: &str) -> Result<Order, ExecutionError>;
    fn open_positions(&self) -> Vec<&Position>;
    fn balance(&self) -> f64;
}
```

```rust
// src/execution/paper.rs — only implementation

```

## Proposed Solution

### Approach

Build a `KrakenTrader` that implements `ExecutionEngine`. Use Kraken's REST API private endpoints with HMAC-SHA512签名. Start with market/limit orders, add stop-loss management, then scale-out logic.

### Steps

1. **Kraken Private API Client** — Add `sign()` method (HMAC-SHA512 + nonce). Add `add_order()`, `cancel_order()`, `get_balance()`, `get_open_orders()`, `get_trade_balance()`.

2. **KrakenTrader struct** — Implement `ExecutionEngine`. Track positions locally. Sync with Kraken on startup.

3. **Order placement** — Market orders for immediate fills. Limit orders for maker-fee routing (0.25% vs 0.40%). Stop-loss as standalone Kraken stop-market orders.

4. **Order status polling** — Poll `OpenOrders` endpoint to detect fills. Update local position state. Handle partial fills.

5. **Balance sync** — Fetch `Balance` endpoint on startup and after each trade. Reconcile with local state.

6. **Error handling** — Retry on transient errors (rate limit, network). Halt on auth errors. Log all API calls.

7. **Engine integration** — `config.mode.paper_trading = false` → use `KrakenTrader`. Keep `PaperTrader` as fallback.

8. **Safety rails** — Max order size cap ($10 at $50 balance). Daily loss halt. Position count limit. All enforced client-side BEFORE sending to Kraken.

9. **Dry-run test** — Place a $1 limit order far from market (won't fill). Verify API auth works. Cancel immediately.

10. **Live test** — Place a $5 market order on BTC/USD. Verify fill. Place stop-loss. Verify stop triggers.

## Perfection Loop

### Loop 1 — RED

1. **No private API client** — KrakenClient only has public endpoints. HMAC-SHA512签名 not implemented.
2. **No order tracking** — PaperTrader tracks orders locally. KrakenTrader needs to poll Kraken for fill status.
3. **No stop-loss on exchange** — PaperTrader simulates stops client-side. KrakenTrader should place stop-market orders on Kraken so stops survive crashes.
4. **No balance sync** — PaperTrader uses hardcoded starting_balance. KrakenTrader needs to fetch real balance.
5. **No error recovery** — What happens if Kraken API is down mid-trade? Need retry + halt logic.
6. **No partial fill handling** — Kraken may partially fill a limit order. Need to track partial quantities.
7. **No rate limit handling** — Kraken private API has 15-second decay rate limits. Need backoff.
8. **No order size validation** — Kraken has minimum order sizes per pair. Need to check before placing.
9. **No fee calculation** — Real fees vary by volume tier. Need to fetch actual fee rate from Kraken.
10. **Engine doesn't switch** — `run()` function always creates PaperTrader. Need conditional creation.
11. **No kill switch** — No way to immediately stop agent + cancel all orders if something goes wrong.
12. **No notifications** — `data/alerts.jsonl` is local only. Need real-time alerts for trade events.
13. **No startup balance check** — Agent doesn't verify Kraken balance before placing orders.
14. **No sandbox testing** — Should test full lifecycle on Kraken sandbox before real money.
15. **No slippage protection** — No alert if fill differs significantly from expected price.
16. **No daily P&L enforcement** — Daily loss limit exists in config but not enforced in KrakenTrader.

### Loop 1 — GREEN

1. **Fix 1:** Add `sign()` to KrakenClient. Implement `add_order()`, `cancel_order()`, `balance()`, `open_orders()`, `trade_balance()`.
2. **Fix 2:** Build `KrakenTrader` with local position tracking + Kraken sync.
3. **Fix 3:** Place stop-market orders on Kraken for every position. Track stop order IDs.
4. **Fix 4:** Fetch `Balance` on startup. Reconcile after each trade.
5. **Fix 5:** Retry with exponential backoff (1s, 2s, 4s, max 3 retries). Halt on 401/403.
6. **Fix 6:** Track `filled_qty` per order. Update position quantity on partial fill.
7. **Fix 7:** Rate limiter: max 1 private API call per 15 seconds. Queue excess requests.
8. **Fix 8:** Fetch `AssetPairs` minimum order size. Reject orders below minimum.
9. **Fix 9:** Fetch `TradeVolume` for actual fee rate. Use in P&L calculations.
10. **Fix 10:** `config.mode.paper_trading` flag controls which implementation to use.
11. **Fix 11:** `cargo run -- kill` command: cancel all open orders, close all positions, halt engine.
12. **Fix 12:** Discord webhook integration for TRADE_OPENED, TRADE_CLOSED, SLIPPAGE_ALERT events.
13. **Fix 13:** Fetch `Balance` on startup. If balance < minimum order size, halt with clear error.
14. **Fix 14:** Kraken sandbox mode via config flag. Test full lifecycle before real money.
15. **Fix 15:** If |fill_price - expected_price| / expected_price > 0.005, log warning + pause 5min.
16. **Fix 16:** Track daily P&L in KrakenTrader. If cumulative loss > 10% of starting balance, cancel all + halt until next UTC day.

### Loop 1 — AUDIT (Five Questions)

1. **ALL cases?** ✅ Every order goes through the same path: validate → sign → send → poll → update. Stop-losses are exchange-side, surviving crashes.
2. **1000 agents?** ✅ KrakenTrader is stateless (positions synced from Kraken). Multiple instances would conflict on order IDs, but we only run one.
3. **Hostile attacker?** ✅ API keys in .env (gitignored). HMAC签名 prevents replay attacks (nonce). No secrets in logs.
4. **2 years?** ✅ Kraken API v0 is stable. Fee rates may change but are fetched dynamically. Order minimums may change but are fetched per-pair.
5. **Industry standard?** ✅ HMAC-SHA512签名 is Kraken's standard. Stop-market orders are standard. Rate limiting is standard.

## Lessons Learned

1. **Paper trading results are not live results.** $12K simulated return assumes perfect fills. Real execution has slippage, partial fills, and latency.
2. **Exchange-side stops are mandatory.** Client-side stops die with the process. Kraken stop-market orders survive crashes.
3. **HMAC签名 is the hard part.** The nonce must be strictly increasing. Clock drift can cause auth failures. Need monotonic counter, not timestamp.
4. **Rate limits are per-endpoint-type.** Orders and balance have separate 15s decay windows. Can't just use a single rate limiter.
5. **Minimum order sizes vary per pair.** BTC/USD might be 0.0001 BTC. DOGE/USD might be 10 DOGE. Must fetch from AssetPairs API.
