## Summary

Knowledge selection is broken due to data quality issues, not architectural
flaws. All 2,959 units have priority 5 (zero signal), 282 execution units have
zero conditions (invisible), 301/350 risk units are catch-all (always match),
and `context_tags` is wired but never populated. The proposed catalog approach
was rejected by Perfection Loop — Trending alone matches 1,879 units, making
the catalog 47K tokens (4x worse than current 11K dump). The correct fix is to
repair the data inputs to the existing MMR system: fix priorities, fix
conditions, wire context_tags from indicators, and cap at 20 units max.

## Environment

- **OS:** Windows (win32)
- **Language/Runtime:** Rust 1.91
- **Tool Versions:** cargo 1.91, reqwest 0.12 (stream feature)
- **Commit/State:** v0.4.2, streaming LLM provider just added

## Detailed Description

### Problem

1. **All priority 5** — Every knowledge unit has `"priority": 5`. The scoring
   formula `condition_score * 3 + priority * 2 + tag_score` reduces to
   `condition_score * 3 + 10 + tag_score`. Priority provides zero signal.

2. **Execution units invisible** — `knowledge_execution.json` has 282 units
   with ZERO conditions. They never match any market state and are never
   selected. 9.5% of knowledge is dead.

3. **Risk units are catch-all** — 301/350 risk units have 5+ conditions
   (ExtremeFear, ExtremeGreed, HighVolatility, LowVolatility + more). They
   ALWAYS match regardless of market state, crowding out relevant units.

4. **Trending matches everything** — 1,879/2,959 units have Trending condition.
   In a trending market, 63% of all knowledge passes the pre-filter. The
   "filter" isn't filtering.

5. **context_tags never populated** — Engine passes `vec![]` to build_context.
   The tag-based matching system (RSI, EMA, ADX, VWAP, VP, OBI tags) is wired
   but produces zero signal.

6. **No unit cap** — `select_with_tags()` has no max count, only token budget.
   With 1,879 matching units and a generous budget, 113 units get dumped.

### Expected Behavior

Knowledge selection should return 15-20 highly relevant units based on the
current market state, regime, indicator readings, and session. Not 113 generic
units where everything is priority 5.

### Root Cause

1. Priority was set to 5 uniformly during the knowledge extraction sprint —
   no differentiation was applied.
2. Conditions were assigned too broadly — risk units got every condition,
   execution units got none.
3. `context_tags` was implemented but never wired from the engine.

## Perfection Loop

### Loop 1 — Catalog Approach REJECTED

**RED — Issues with original catalog design:**

1. **Token bomb**: Trending matches 1,879 units. Catalog at 100 chars/unit =
   188K chars = ~47K tokens. Current dump is ~11K. Catalog is 4x WORSE.

2. **Two-pass latency**: Pre-selection call (30-90s) + decision call (30-90s) =
   60-180s per pair. With 2 pairs on 300s interval, barely fits.

3. **ID parsing fragile**: LLMs don't naturally output structured references
   like "ta-001". Parsing would need fuzzy matching, high false positive rate.

4. **Execution units still invisible**: Catalog inherits the same broken
   condition filter. 282 units still never appear.

5. **Over-engineering**: The MMR system is architecturally sound. The inputs
   are broken. Fixing inputs is simpler and more effective than replacing
   the system.

**GREEN — Revised approach: Fix the data, not the architecture.**

1. **Fix priorities** — Assign 1-5 based on specificity:
   - 5: Specific setups with clear triggers and price levels
   - 4: Strong tactical knowledge with conditions
   - 3: General strategy knowledge
   - 2: Foundational/conceptual (Dow Theory, market structure)
   - 1: Background/contextual
   
2. **Fix conditions** — Repair condition assignments:
   - Add conditions to execution units (282 currently invisible)
   - Remove catch-all conditions from risk units (keep 2-3 most relevant)
   - Add indicator-derived conditions (see step 3)

3. **Add indicator-derived conditions** — New conditions in
   `determine_conditions()`:
   - `Oversold` (RSI < 30), `Overbought` (RSI > 70)
   - `StrongTrend` (ADX > 25), `WeakTrend` (ADX < 20)
   - `VolumeExpansion` (volume > 1.5x 20-period SMA)
   - `TrendAlignment` (EMA fast > EMA slow)
   
4. **Wire context_tags** — Populate from engine indicators:
   - RSI → "oversold" / "overbought" / "neutral"
   - ADX → "strong_trend" / "weak_trend"
   - EMA → "crossover_bull" / "crossover_bear"
   - VWAP → "above_vwap" / "below_vwap"
   - OB imbalance → "bid_heavy" / "ask_heavy"

5. **Cap at 20 units** — Hard limit in `select_with_tags()`. MMR already
   handles diversity. 20 units × 400 chars = 8K chars = ~2K tokens.

**AUDIT — Five Questions on revised design:**

1. ALL cases? ✅ Indicator conditions cover RSI/EMA/ADX/volume extremes.
   Falls back to regime + fear/greed when no extremes fire.
2. 1000 agents? ✅ Stateless. Same files, same logic.
3. Hostile attacker? ✅ Read-only knowledge, no user input in selection.
4. 2 years? ✅ Additive changes. Priority/conditions are metadata.
5. Industry standard? ✅ MMR is well-studied for IR. Indicator conditions
   are standard in algo trading.

## Lessons Learned

1. **Don't replace what's broken — fix what's broken.** The MMR architecture
   is sound. The data inputs were garbage. Catalog approach was rejected
   because it inherited the same broken filter.

2. **Token budget must be calculated against worst-case data.** 1,879 units
   matching Trending × 100 chars = 47K tokens. Design must account for the
   maximum, not the average.

3. **Assign priorities during extraction, not after.** Priority 5 for
   everything is the same as priority 1 for everything. The extraction sprint
   should have included a specificity assessment.

4. **Catch-all conditions are worse than no conditions.** Risk units matching
   EVERY state means they crowd out actually relevant knowledge in every call.

5. **Wiring is not the same as working.** `context_tags` was wired into the
   API but never populated from the engine. Compilation ≠ verification.
