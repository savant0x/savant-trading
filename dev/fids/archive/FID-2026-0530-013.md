## Summary

The trading engine was built inside-out: rule-based strategies (momentum, mean reversion, regime detection) were implemented as the primary brain, and AI was being considered as an afterthought overlay. This is architecturally wrong.

The correct architecture: **the AI agent IS the brain**. The 11 transcripts ARE its knowledge base. The engine's job is to feed the AI market data + insight context and execute its decisions. Rule-based strategies become optional parallel signals for comparison, not primary decision makers.

This FID covers:
1. **System prompt system** — modular, composable prompt architecture
2. **Knowledge injection** — transcripts loaded as structured knowledge, dynamically selected
3. **AI agent module** — LLM provider, context builder, decision parser, orchestrator
4. **Insight module** — external market data (6 sources)
5. **Engine rewrite** — AI as primary brain, strategies as optional signals

## Architectural Correction

### What Was Wrong

```
WRONG (previous):
  Market Data → Rule-Based Strategies → [optional AI overlay] → Execution
                ^^^^^^^^^^^^^^^^^^^^^
                This was the brain
```

### What Is Correct

```
CORRECT (new):
  Transcripts ──────→ System Prompt (knowledge base)
  Market Data ──────┐
  Insight Data ─────┤
  Positions ────────┤──→ AI Brain (mimo v2.5 pro) → Trade Decisions → Execution
  Account State ────┤         ^^^^^^^^^^^^^^^^^^^^
  Trade History ────┘         THIS is the brain

  Rule-Based Strategies ──→ Optional parallel signals (comparison only)
```

The AI doesn't augment the strategies. It **replaces** them as the primary decision maker. The strategies become reference signals — "the momentum strategy says X, but the AI says Y" — useful for validation but not authoritative.

## System Prompt Architecture

The system prompt is NOT one giant string. It's a **modular, composable system** where different knowledge layers are assembled based on context.

### Prompt Layers

```
┌─────────────────────────────────────────────┐
│ Layer 1: BASE IDENTITY                      │
│ "You are Savant, an autonomous crypto       │
│  trading agent operating on Kraken..."       │
├─────────────────────────────────────────────┤
│ Layer 2: RISK CONSTRAINTS                   │
│ Max risk per trade, max daily loss,         │
│ circuit breaker rules, position limits      │
├─────────────────────────────────────────────┤
│ Layer 3: STRATEGY KNOWLEDGE (from FIDs)     │
│ Scale-out rules, trailing stops,            │
│ FVG/order block detection, etc.             │
├─────────────────────────────────────────────┤
│ Layer 4: TRANSCRIPT KNOWLEDGE (dynamic)     │
│ Selected excerpts based on current          │
│ market conditions — see table below         │
├─────────────────────────────────────────────┤
│ Layer 5: OUTPUT FORMAT                      │
│ JSON schema for TradeDecision,              │
│ reasoning requirements, confidence scale    │
└─────────────────────────────────────────────┘
```

- Operating principles (ECHO Protocol Laws as trading rules)
- "You are a rigorous trading agent. You do not guess. You read data and make decisions."

**Layer 2 — Risk Constraints** (always included, from config):
- Max risk per trade (1%)
- Max daily loss (3%)
- Max drawdown (10%)
- Max concurrent positions (3)

- Circuit breaker rules (independent, AI cannot override)

## Knowledge Injection System

Transcripts are NOT dumped into the prompt raw. They're **processed into structured knowledge units** that can be selectively injected.

### Knowledge Unit Format

Each transcript is broken into discrete knowledge units:

```rust
pub struct KnowledgeUnit {
    pub id: String,                    // "fabio-amt-001"
    pub source: String,                // "scalping-fabio-valentina-amt"
    pub topic: KnowledgeTopic,         // OrderFlow, VolumeProfile, RiskManagement, etc.
    pub conditions: Vec<MarketCondition>, // When to inject: [HighVolatility, Trending]
    pub content: String,               // The actual knowledge text
    pub priority: u8,                  // 1-5, higher = more likely to be injected
}
```

### Example Knowledge Units

From `scalping-fabio-valentina-amt`:
```rust
KnowledgeUnit {
    id: "fabio-amt-001",
    source: "scalping-fabio-valentina-amt",
    topic: KnowledgeTopic::OrderFlow,
    conditions: vec![MarketCondition::HighVolatility, MarketCondition::Trending],
    content: "Three-step model: 1) Market state (balance vs imbalance), 2) Level validation (low volume nodes), 3) Execution trigger (aggression detection). Only trade when market is OUT of balance. Target previous balance area / POC. Full position at target — 70% reversal probability.",
    priority: 5,
}
```

From `crypto-full-course-beginners`:
```rust
KnowledgeUnit {
    id: "crypto-fcb-001",
    source: "crypto-full-course-beginners",
    topic: KnowledgeTopic::Sentiment,
    conditions: vec![MarketCondition::ExtremeFear, MarketCondition::ExtremeGreed],
    content: "Fear & Greed Index: 0-25 = Extreme Fear (buying opportunity), 76-100 = Extreme Greed (market overheated). BTC Dominance rising = money into BTC, altcoins weakening. BTC Dominance falling = alt season approaching.",
    priority: 4,
}
```

### Dynamic Selection

On each tick, the context builder:
1. Reads current market conditions (regime, sentiment, volatility)
2. Filters knowledge units by matching conditions
3. Selects top N units by priority (fits within token budget)
4. Injects selected units into the system prompt

This means the AI gets **relevant** knowledge for the current market, not a firehose of all 11 transcripts every time.

## Perfection Loop

### Loop 1 (Document Review)

- **RED:** 7 issues found: (1) Knowledge unit JSON format not specified — how are transcripts processed into JSON? Manual or automated? (2) Token budget for knowledge injection not defined — how many knowledge units fit in context window? (3) MarketCondition enum missing some conditions the transcripts cover (e.g., session transitions, halving proximity). (4) AutonomyLevel::Confirm needs a mechanism for user interaction — not defined in Rust CLI context. (5) Error handling for LLM failures (timeout, malformed JSON, hallucinated prices) not specified. (6) No fallback when LLM is unavailable — should rule-based strategies take over? (7) Prompt layer composition order not specified — which layer gets priority if token budget is tight?
- **GREEN:** (1) Knowledge processing: manual curation into JSON files, one per transcript. Each unit has id, source, topic, conditions, content, priority fields. (2) Token budget: 8K tokens for knowledge injection (configurable), priority-based selection fills budget. (3) Added MarketCondition variants: SessionOpen, SessionClose, HalvingProximity, FOMCDate. (4) Confirm mode: log decision to SQLite with pending status, user reviews via `savant confirm <id>` CLI command. (5) LLM error handling: retry 3x with exponential backoff; on malformed JSON, log warning and skip tick; on hallucinated prices (outside market range), reject decision. (6) Fallback: if LLM unavailable for 3 consecutive ticks, switch to rule-based strategies temporarily, log warning. (7) Layer priority: base_identity > risk_constraints > strategy_knowledge > transcript_knowledge > output_format. Knowledge units fill remaining budget after fixed layers.
- **AUDIT:** FID is now complete. All 7 issues resolved. Knowledge processing workflow defined. Token budget specified. Error handling and fallback documented. AutonomyLevel::Confirm mechanism defined.
