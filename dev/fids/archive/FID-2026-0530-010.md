## Summary

TJR's core teaching is 4 timeframes for confluence: execution (5m), structure (15m), bias (1h), trend (4h). Currently the engine only uses one timeframe. Signals generated on a single timeframe lack the higher-timeframe context that filters out low-probability trades.

## Detailed Description

### Problem

`Strategy::evaluate()` takes `&[Candle]` from a single timeframe. No higher-timeframe data available. A bullish 5m signal might conflict with a bearish 4h trend.

### Expected Behavior

- Fetch candles on multiple timeframes (5m, 15m, 1h, 4h)
- Strategy evaluates higher timeframe for bias before generating signal
- Only generate signal if higher timeframe aligns (e.g., 5m long only if 1h is also bullish)

### Evidence

```
strategy/base.rs — evaluate() takes single &[Candle]
engine.rs — fetches one timeframe
config.rs — single `timeframe` field
```

## Impact Assessment

### Affected Components

- `src/core/config.rs` — add `timeframes: Vec<String>` to `TradingConfig` (replaces single `timeframe`)
- `src/core/types.rs` — add `Timeframe` to `IndicatorValues` or create `MultiTimeframeData` struct
- `src/strategy/base.rs` — change `evaluate()` signature to accept multiple timeframe data
- `src/strategy/momentum.rs` — check higher timeframe trend before generating signal
- `src/strategy/mean_reversion.rs` — check higher timeframe bias
- `src/engine.rs` — fetch candles on all configured timeframes
- `src/data/market_data.rs` — store candles per timeframe per pair

### Risk Level

- [x] Medium: Signals lack higher-timeframe context

## Proposed Solution

### Steps

1. **Config** — Replace `timeframe: String` with `timeframes: Vec<String>` (default: `["5m", "15m", "1h", "4h"]`). Add `execution_timeframe: String` (default: "5m").
2. **Multi-timeframe store** — `HashMap<String, HashMap<String, MarketDataStore>>` (pair → timeframe → store)
3. **Strategy signature** — Change `evaluate()` to:
   ```rust
   async fn evaluate(
       &self,
       candles: &[Candle],                    // execution timeframe
       higher_tf_candles: &[Candle],          // highest non-execution timeframe
       indicators: &IndicatorValues,
       regime: MarketRegime,
       volume_profile: Option<&VolumeProfile>,
   ) -> Option<Signal>;
   ```
4. **Higher TF filter** — In momentum strategy, only generate long signal if higher TF EMA is bullish (price > EMA). In mean reversion, only generate signal if higher TF is ranging (ADX < threshold).
5. **Engine loop** — Fetch all timeframes, pass execution + highest to strategy
6. **Verify** — build + clippy + fmt

### Verification

- Verify candles fetched on all 4 timeframes
- Verify signals only fire when higher timeframe aligns
- Verify no signals when higher timeframe conflicts
