## Summary

The codebase supports Kraken (CEX, requires KYC) and PaperTrader (simulated). The user's ID is expired and cannot verify their Kraken account, but has $50 in ETH on MetaMask. We need to add a **DEX execution backend** that works without KYC via 0x and 1inch APIs on Arbitrum (low gas), using the wallet's private key to sign and broadcast transactions.

## Detailed Description

### Problem

Only two execution backends exist: `PaperTrader` (simulated) and `KrakenTrader` (requires verified Kraken account with KYC). The user cannot use Kraken and has no way to execute real trades. A DEX aggregator backend on Arbitrum would allow:

1. No KYC ‚Äî trade directly from self-custodial wallet
2. Low gas fees ‚Äî $0.02-0.10 per trade on Arbitrum
3. $50 viable ‚Äî gas doesn't eat the position
4. Same `ExecutionEngine` interface ‚Äî minimal changes to the AI agent

### Expected Behavior

A `DexTrader` implementing `ExecutionEngine` that:

1. Accepts a configurable backend (`0x` or `1inch`) via `exchange.backend` config field
2. Resolves trading pairs (e.g., "ETH/USDC") to token addresses on Arbitrum
3. Calls the DEX API to get swap quotes and execution calldata
4. Signs transactions with the wallet private key
5. Broadcasts to Arbitrum via RPC
6. Tracks positions locally with client-side stop monitoring
7. Reports P&L from fill prices and gas costs

### Root Cause

No DEX integration exists. The execution layer only knows CEX (Kraken) and simulated (PaperTrader).

### Position Management

Since DEXes don't have stop-loss orders, the `DexTrader` monitors stops client-side (same pattern as `PaperTrader.check_stops()`). This means:
- Stops checked on every tick
- If stop is hit, a market close swap is executed
- Positions tracked locally in `HashMap<String, Position>`
- Balance = wallet's token balance (USDC for base currency)

## Docs Audit Findings

After the initial implementation, the user requested a **RED PHASE** verification against actual API documentation. The audit uncovered:

### 0x API ‚Äî 4 issues found & fixed
| Issue | Claimed | Actual (from docs) |
|-------|---------|--------------------|
| Endpoint version | `/swap/v1/quote` | **`/swap/v2/quote`** |
| Query param name | `takerAddress` | **`taker`** |
| Missing param | ‚Äî | **`chainId`** required in v2 |
| Response gas field | Flat `gasPrice` only | **`gasFees` object** with `maxFeePerGas` |

### 1inch API ‚Äî 2 issues found & fixed
| Issue | Claimed | Actual (from docs) |
|-------|---------|--------------------|
| Quote request | Included `walletAddress` param | **Quote is read-only** ‚Äî no wallet param |
| Slippage format | `(round() as u64)` integer | **Float string** (e.g. `"0.5"` for 0.5%) |

### EIP-1559 Signing ‚Äî 2 critical bugs found & fixed
| Bug | Description | Impact |
|-----|-------------|--------|
| Missing `0x02` prefix in signing hash | `keccak256(rlp(...))` computed without type byte | All transactions would be **rejected on-chain** ‚Äî wrong signature |
| Access list as `0x80` instead of `0xc0` | `Vec::new()` RLP-encodes as empty byte string, not empty list | All transactions would **fail RLP decoding** on the node |

### Lesson: Follow RED PHASE protocol before writing code

Always pull the actual documentation for every external API and specification before implementing. The 0x API had migrated from v1 to v2, and the EIP-1559 signing spec had subtle details that were easy to miss.

## Lessons Learned

1. DEX trading has fundamentally different risk profile from CEX ‚Äî no exchange-side stops, gas costs, MEV exposure
2. 0x API requires API key even for basic quotes (free tier available). API v1 is deprecated ‚Äî **v2 is required** with `chainId` query param and `taker` (not `takerAddress`).
3. 1inch API v6.1 uses float strings for slippage (e.g. `"0.5"` = 0.5%) and `/quote` is read-only (no `from` address).
4. EIP-1559 signing is subtle: the `0x02` type prefix is part of the **signing hash**, not just the wire format. And the access list field must be an RLP **list** (`0xc0`), not an RLP **byte string** (`0x80`).
5. Arbitrum gas is low enough that $50 trading is viable
6. Wallet private key security must follow same pattern as Kraken API keys (env var only, never logged)
7. Always pull actual docs before implementing ‚Äî assumptions about API endpoints, param names, and response shapes are unreliable
