## Summary

Seven issues found during first live paper trading run: (1) sequential AI evaluation takes 7.5-15 min per cycle for 15 pairs — must be parallelized to ~30s; (2) Hold decisions are not logged to vault or API — no visibility into AI reasoning; (3) AI returns Hold for all 15 pairs in Extreme Fear market — prompt needs optimization; (4) order book imbalance data fetched but never passed to AI; (5) session info not included in AI context; (6) no per-call LLM timeout for parallel evaluation; (7) no error isolation — one failed LLM call aborts the whole batch. All seven must be fixed before the bot can function as intended.

---

## Issues

### CRIT-1. Sequential AI evaluation — 15 pairs take 7.5-15 min per cycle

**File:** `src/engine.rs:356-714`
**Problem:** The engine loop evaluates pairs sequentially: `for pair in &config.trading.pairs { ... agent.evaluate(&ctx).await ... }`. Each LLM call takes 30-60s. 15 pairs = 7.5-15 min per cycle. With 5-min candles, the engine misses 1-3 cycles per evaluation.
**Root Cause:** The loop structure processes one pair at a time through the full pipeline (fetch → indicators → AI → execute). The AI call is the bottleneck.
**Fix:** Restructure into 3 phases:
- Phase 1 (sequential, fast): Fetch all candles + order books + compute indicators for all pairs
- Phase 2 (parallel, slow): Send all 15 LLM calls simultaneously via `tokio::JoinSet` or `futures::future::join_all`
- Phase 3 (sequential, fast): Process all results — execute trades, update positions, check stops

The LLM provider (`self.provider.chat()`) is stateless — just an HTTP POST. It can be called concurrently. The agent's fallback tracking can be updated after all results return.

**Verification:** `cargo run` — log shows all 15 "ai_context" lines within 2 seconds of each other, then all 15 AI DECISION lines within ~60 seconds total.
**Lines changed:** ~120 (major loop restructuring)

### CRIT-2. Hold decisions not logged — no visibility into AI reasoning

**File:** `src/engine.rs:608-619`
**Problem:** When AI returns `Hold`, the engine only checks rule-based strategies for comparison. The Hold decision (with reasoning) is not logged to vault, not stored in `SharedEngineData.decisions`, not visible via API.
**Root Cause:** The `AgentResult::Hold` match arm has no logging or storage.
**Fix:**
1. In orchestrator: return `AgentResult::Decision(decision)` even for Hold actions (not just `AgentResult::Hold`)
2. In engine: log ALL decisions (including Hold) to `shared.decisions` and vault
3. Add `decision.action == Hold` check to skip execution but still log

**Verification:** `curl localhost:8080/api/decisions` shows Hold decisions with reasoning. Vault `Decisions/` folder has Hold entries.
**Lines changed:** ~30

### CRIT-3. AI returns Hold for all 15 pairs — prompt needs optimization

**File:** `src/agent/prompts.rs`, `src/agent/context_builder.rs`
**Problem:** In Extreme Fear (23) market, the AI returns Hold for all 15 pairs. With 254 knowledge units covering contrarian strategies (buy fear, sell greed), the AI should be finding setups. The prompt doesn't emphasize that Extreme Fear IS a setup condition.
**Root Cause:** The prompt tells the AI to "analyze and decide" but doesn't explicitly connect market conditions to knowledge units. The AI sees "Extreme Fear" in the context and "buy fear" in the knowledge, but doesn't bridge them.
**Fix:**
1. Add explicit condition-to-action mapping in the system prompt: "When Fear & Greed < 25, you are in a BUYING ZONE. Look for long entries on pairs with strong support."
2. Add "You must take action on at least 2-3 pairs per cycle unless ALL pairs show no setup" — prevents blanket Hold
3. Ensure the knowledge selection is actually injecting fear-related units when Fear & Greed is extreme

**Verification:** Next run produces at least 1 Buy or Sell decision across 15 pairs.
**Lines changed:** ~20

### CRIT-4. Order book imbalance not in AI context

**File:** `src/agent/context_builder.rs`, `src/engine.rs:382-404`
**Problem:** The engine fetches order book data every tick and logs imbalances (bid-heavy/ask-heavy), but this data is never passed to the AI. The `FullContext` struct has no order book field. The AI is making decisions without seeing real-time supply/demand pressure.
**Root Cause:** Order book was wired for logging but not for AI consumption.
**Fix:**
1. Add `order_book_imbalance: Option<f64>` to `FullContext`
2. Populate from `ob_manager.imbalance(5)` in the engine loop
3. Include in `build_user_message_static`: "Order Book Imbalance: +0.45 (bid-heavy — buying pressure)"
4. Pass imbalance to orchestrator context

**Verification:** AI context log shows "Order Book Imbalance" line.
**Lines changed:** ~15

### CRIT-5. Session info not in AI context

**File:** `src/agent/context_builder.rs`
**Problem:** The session multiplier is applied to position sizing in the engine, but the AI doesn't know what session it's in. The knowledge base has session-specific strategies (Juvier's kill zones, session highs/lows) but the AI can't use them without knowing the current session.
**Root Cause:** Session detection exists (`core::session::current_session()`) but is only used for position sizing, not injected into AI context.
**Fix:**
1. Add `session: Session` to `FullContext`
2. Populate from `current_session()` in engine loop
3. Include in `build_user_message_static`: "Current Session: London (2 AM - 5 AM EST) — High volume, reversals expected. Kill zone active."
4. This enables the AI to apply session-specific knowledge units

**Verification:** AI context log shows "Current Session: London" with behavior description.
**Lines changed:** ~10

### CRIT-6. No per-call LLM timeout — parallel evaluation hangs if one call stalls

**File:** `src/agent/provider.rs`, `src/agent/orchestrator.rs`
**Problem:** With parallel evaluation, if one LLM call hangs (OpenGateway timeout, network issue), it blocks all 15 results. The `reqwest` client has no per-request timeout configured. A single hung call can stall the entire cycle indefinitely.
**Root Cause:** No timeout on the HTTP client or the `chat()` call.
**Fix:**
1. Add `timeout_secs: u64` to `LlmConfig` (default: 90)
2. Configure `reqwest::Client::builder().timeout(Duration::from_secs(timeout_secs))`
3. In parallel evaluation, use `tokio::time::timeout(Duration::from_secs(90), call)` wrapper
4. On timeout: log warning, return Hold for that pair, continue processing other pairs

**Verification:** If OpenGateway hangs for one pair, other pairs still complete and the cycle finishes.
**Lines changed:** ~15

### CRIT-7. No error isolation — one failed LLM call aborts the whole batch

**File:** `src/engine.rs` (parallel evaluation section)
**Problem:** When using `join_all` or `JoinSet` for parallel LLM calls, if one call panics or returns an error, the behavior depends on the collection method. `join_all` will still collect all results but `JoinSet` may abort. Need explicit error isolation.
**Root Cause:** No error handling strategy for parallel evaluation failures.
**Fix:**
1. Use `tokio::JoinSet` with explicit error handling per task
2. Each parallel task wraps its LLM call in `tokio::time::timeout` + `Result` handling
3. On error: log warning, return `AgentResult::Hold` for that pair
4. Collect all results regardless of individual failures
5. Log summary: "14/15 pairs evaluated successfully, 1 timed out"

**Verification:** Kill OpenGateway mid-cycle — remaining pairs still complete, engine logs which pairs failed.
**Lines changed:** ~25
