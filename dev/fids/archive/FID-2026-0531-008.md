## Detailed Recommendations

### 1. Forward-Simulated Returns + Virtual Wallet
**Impact: HIGHEST** — Currently we grade decisions but don't simulate outcomes or track P&L.

**Virtual Wallet:**
- Starting balance: $50 (matches config.trading.starting_balance)
- Track: balance, equity, unrealized P&L, realized P&L, open positions, closed positions
- Fee model: 0.26% per trade (Kraken taker fee)
- Slippage model by asset tier:
  - BTC/ETH normal: 0.05%, high vol: 0.10-0.15%
  - Top 10 alts: 0.10-0.20%
  - Top 100 alts: 0.30-0.50%
  - Microcaps: 1.00-5.00%

**Trade Simulation:**
- Decision happens at candle N, simulate from N+1 to end of candle data
- Track MFE (Maximum Favorable Excursion) and MAE (Maximum Adverse Excursion)
- Partial fills: 30% at TP1, 30% at TP2, 40% at TP3
- Pessimistic execution: if stop and TP hit same candle, assume stop hit first
- Calculate true R-multiple per trade

**Portfolio Metrics (per sandbox run):**
- Total P&L ($ and %)
- Win rate (%)
- Average R-multiple
- Max drawdown ($ and %)
- Sharpe ratio (if enough trades)
- Profit factor (gross profit / gross loss)
- Equity curve data points (for charting)

### 1b. Run Reports (Persistent Logs)
**Impact: HIGH** — Currently sandbox output is printed to stdout and lost. Need persistent, structured reports.

**Report Structure:**
Each sandbox run generates a timestamped report at `data/sandbox_reports/YYYY-MM-DD_HH-MM-SS.md` containing:

1. **Header** — Run ID, timestamp, config snapshot (pairs, timeframe, concurrency, API endpoint, model)
2. **Wallet Summary** — Starting/final balance, P&L, win rate, R-multiple, max drawdown, profit factor
3. **Trade Log** — Every trade with: scenario, action, entry, stop, TP1/2/3, exit price, P&L, R-multiple, hold duration, fees paid, slippage applied
4. **Decision Log** — EVERY decision the AI made, full detail:
   - Scenario ID, name, category, difficulty
   - Action taken (Buy/Sell/Hold) with side
   - Entry price, stop loss, TP1/2/3
   - Confidence score
   - Full reasoning text (not excerpt — the complete LLM output)
   - Expected action vs actual action (pass/fail)
   - Grade breakdown (T1 compliance, T2 R:R, T3 reasoning, total)
   - Market conditions at decision time (regime, indicators, sentiment)
   - Knowledge units injected (IDs + topics)
   - Context tags active at decision time
5. **Category Breakdown** — Per-category: passed/total, avg score, avg R-multiple, best/worst trade
6. **Failure Analysis** — ParseErrors, missed trades, low-score scenarios with reasons
7. **Knowledge Stats** — Units injected per scenario, topic distribution, tag match rate
8. **Performance Stats** — Total LLM calls, avg latency, timeout count, error count

**Also generates:**
- `data/sandbox_reports/latest.md` — symlink/copy of most recent report (for quick access)
- `data/sandbox_reports/equity_curve.json` — array of {timestamp, equity} for charting

**API Endpoint:**
- `GET /api/sandbox/reports` — list all reports
- `GET /api/sandbox/reports/latest` — get latest report
- `GET /api/sandbox/reports/:id` — get specific report

### 2. Fault-Tolerant JSON Parsing
**Impact: HIGH** — Currently 8/60 scenarios fail with ParseErrors.

- Use `json-repair` crate for multi-pass heuristic reconstruction
- Pipeline: strict parse → markdown strip → quote normalization → delimiter insertion → truncation recovery
- Expected: ParseErrors from ~13% to near zero
- Partial JSON recovery: extract valid action/entry/stop even from truncated responses

### 3. Multi-Run Averaging with α-Trimmed Mean
**Impact: HIGH** — Single runs are statistically insignificant.

- Run each scenario 5 times
- α-trimmed mean (20% trim): sort scores, drop min and max, average middle 3
- Formula: trimmed_mean = (1/(n-2k)) × Σ x_(k+i) for i=1 to n-2k
- Resistant to outlier hallucinations and lucky runs

### 4. Expected Value (EV) Execution Framework
**Impact: HIGHEST** — Resolves the "Hold everything" problem.

- Transition from "LLM-as-Executor" to "LLM-as-Context-Provider"
- LLM returns probability of success (p) + proposed entry/exit params
- Rust engine independently calculates: EV = (p × reward) - ((1-p) × risk)
- If EV > 0 (after slippage/fees), engine FORCES execution regardless of LLM doubt
- Grades LLM on accuracy of p-estimation and resulting EV, not arbitrary compliance

### 5. MMR Knowledge Selection (Maximal Marginal Relevance)
**Impact: HIGH** — Fixes P5 dominance (Psychology 53%, Risk 64%).

- Replace linear scoring with MMR algorithm
- Formula: MMR = λ × sim(d, q) - (1-λ) × max(sim(d, d_j)) for d_j in selected
- λ = 0.5 balances relevance vs diversity
- Prevents echo chamber: if Risk Management P5 already selected, next Risk P5 gets penalized for similarity
- Forces diversification across Technical, Microstructure, On-Chain, etc.

### 6. Programmatic Context Tags
**Impact: MEDIUM** — context_tags is always vec![].

- RSI divergence detection: price lower low + RSI higher low = bullish divergence tag
- Volume Profile VA: calculate POC, VAH, VAL, emit tags when price interacts
- Order Book Imbalance: (bid_vol - ask_vol) / (bid_vol + ask_vol), threshold ±0.60

### 7. WebSocket Shared State (tokio::sync::watch)
**Impact: MEDIUM** — WS messages queue during LLM inference.

- Replace mpsc channel with watch channel (only retains last value)
- Background task: WS → parse → update Arc<RwLock<MarketState>>
- Inference tasks: watch_rx.changed().await → read lock → build context
- Guarantees zero-latency, always fresh data

### 8. Crypto-Native On-Chain Thresholds
**Impact: MEDIUM** — Agent ignores on-chain signals.

- STH-MVRV < 1.0 = capitulation buy (all short-term holders underwater)
- STH-MVRV > 2.0 = distribution zone (unsustainable unrealized profits)
- STH-SOPR < 1.0 then V-recovery above 1.0 = high-conviction buy
- NVT Golden Cross: short-term MA crosses above long-term MA = overbought

## Steps (dependency-ordered)

### Phase A: Foundation (sequential — each depends on previous)
A1. **Add `json-repair` crate** — Implement `robust_llm_parse()` in `decision_parser.rs`. Pipeline: strict parse → markdown strip → quote normalization → truncation recovery. Target: 0 ParseErrors.
A2. **Build forward simulator + virtual wallet** — New module `src/sandbox/simulator.rs`. SimStruct with starting_balance, positions, trades. For Buy/Sell decisions: simulate trade through remaining candles, track MFE/MAE, partial fills at TP levels, pessimistic stop-first execution. For Hold decisions: no trade simulated, counted as "opportunity cost" in report. Calculate all portfolio metrics. Output wallet report.
A3. **Build run report generator** — New module `src/sandbox/run_report.rs`. Timestamped markdown reports at `data/sandbox_reports/`. Report ID = timestamp (e.g., `2026-05-31_18-15-00`). Includes wallet summary, trade log, decision log, category breakdown, failure analysis, knowledge stats, performance stats. Also generates `latest.md` (copy) and `equity_curve.json`. Wire API endpoints: `GET /api/sandbox/reports`, `GET /api/sandbox/reports/latest`, `GET /api/sandbox/reports/:id`.
A4. **Build EV framework** — LLM returns probability p. Rust calculates EV = (p × reward) - ((1-p) × risk). If EV > 0 after fees/slippage, force execution. Grade on p-accuracy.

### Phase B: Intelligence (parallel — no dependencies between them)
B1. **Implement MMR** — Replace linear scoring in `knowledge.rs`. Use topic+tags overlap as cosine similarity proxy. λ=0.5.
B2. **Build context tag generator** — New module `src/agent/tags.rs`. RSI divergence, Volume Profile VA, OBI from indicators.
B3. **Add on-chain thresholds** — Update SOUL.md with exact STH-MVRV, STH-SOPR, NVT numbers.

### Phase C: Quality (depends on Phase A being fast enough)
C1. **Multi-run averaging** — Each run gets its own independent wallet. 5 runs per scenario, α-trimmed mean on scores. Wallet metrics (P&L, win rate, R-multiple) averaged across 5 runs with same trim. ~300 LLM calls at 10 concurrent = ~15 min.
C2. **Difficulty weighting** — Weight final score by scenario difficulty: Easy=1.0, Medium=1.5, Hard=2.0, Extreme=3.0. Harder scenarios count more in the aggregate score. Prevents the score from being propped up by easy passes.
C3. **Always-hold benchmark** — Run the sandbox once with a hardcoded "Hold" decision for every scenario. Grade it. This proves the AI adds value vs doing nothing. If AI score < always-hold score, something is fundamentally wrong.
C4. **Regression testing** — After each sandbox run, compare score against last 3 runs (stored in `data/sandbox_reports/`). If score drops >5% vs rolling average, print a REGRESSION WARNING with the specific categories that degraded.

### Phase D: Production (separate FID — not sandbox)
D1. **WebSocket shared state** — tokio::sync::watch + Arc<RwLock<MarketState>>. Production engine change, not sandbox.
