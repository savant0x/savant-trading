## Perfection Loop

### RED — All Issues

1. **Semantic consolidation** — `semantic_patterns` table empty. No SQL
   aggregations to populate it.
2. **Anti-pattern detection** — No queries for failure conditions. No
   injection into prompts.
3. **GEPA mutation** — feedback.rs generates proposals but never applies them.
   No Teacher LLM. No pareto gatekeeper. No auto-rollback.
4. **Knowledge lifecycle** — `knowledge_units_used` always empty. No utility
   scoring. No pruning.
5. **Held-out validation** — All 60 scenarios used for everything. No train/val
   split. Overfitting guaranteed.
6. **SOUL.md version control** — No `soul_versions` table. Mutations not
   tracked. No rollback capability.
7. **Reflexion replay** — replay.rs queries losses but doesn't auto-generate
   lessons via Teacher LLM.
8. **Scenario expansion** — No mechanism to convert live failures into
   permanent scenarios.
9. **Multi-asset correlation** — BTC/USD and ETH/USD 90% correlated. Opening
   both = 2x risk. Agent doesn't know.
10. **Portfolio heat** — max_positions=5 but each could risk 10%. Total heat
    could be 50%. Circuit breaker doesn't check this.
11. **SQLite backup** — SQLite WAL is robust but not corruption-proof. Brain
    loss = total loss.
12. **WS backoff** — Fixed 5s reconnection. No exponential backoff. Rapid
    reconnection storms possible.
13. **Dynamic slippage** — Fixed slippage_pct. Doesn't account for order book
    depth, volatility, or order size.

### GREEN — All Solutions

**Phase 1: Data Foundation**

1. **Semantic consolidation** — Create `memory/semantic.rs`. SQL aggregations
   against `agent_episodes` + `episode_market_context`:
   - Regime/Session/Pair edge matrix (win_rate, avg_rr, profit_factor)
   - Conviction calibration matrix (confidence bucket vs actual outcomes)
   - Knowledge unit efficacy (which units correlate with wins)
   - Rolling 90-day window for pattern decay
   - Minimum N >= 5 threshold (increase to 10 after 100 episodes)
   - Populate `semantic_patterns` table

2. **Anti-pattern detection** — Create `memory/anti_pattern.rs`. SQL queries:
   - Conditions where win_rate < 30% AND N >= 5
   - Multidimensional: funding + RSI + regime combinations
   - Format as narrative prompt injection (not hard blocks)
   - Example: "When Funding > 0.05% and RSI > 70, your PF is 0.32 (N=24)"

3. **Wire into context_builder** — Add semantic patterns + anti-patterns to
   the 6th prompt layer (Dynamic Memory Context). Inject after market data,
   before decision request.

**Phase 2: Risk Model**

4. **Multi-asset correlation** — Add `risk/correlation.rs`. Rolling 48-period
   Pearson correlation between active pairs' candle returns. Store as
   correlation matrix. Adjust effective position count:
   - Correlation < 0.5: count as 1.0 positions
   - Correlation 0.5-0.8: count as 1.5 positions
   - Correlation > 0.8: count as 2.0 positions
   - Modify circuit_breaker.rs to use effective positions

5. **Portfolio heat** — Add to circuit_breaker.rs. Calculate:
   `heat = sum(position.risk_amount) / account.equity`
   Block new trades if heat > 40%. Log heat in status output.

6. **Dynamic slippage** — Modify paper.rs `place_order()`:
   `slippage = base_slippage + (order_value / book_depth) * atr_multiplier`
   Where book_depth = sum of top 5 order book levels, atr_multiplier scales
   with current ATR vs average ATR. Need order book data passed to place_order.

**Phase 3: Knowledge Lifecycle**

7. **utility_score field** — Add to KnowledgeUnit (serde default 1.0). Adjust
   MMR scoring: `final_score = mmr_score * (1.0 + utility_score.log2())`.

8. **knowledge_units_used capture** — Wire into episode snapshot. Query from
   build_context output (the selected units list).

9. **Utility update batch job** — After each training run: for each episode,
   increment utility_score of used knowledge units if is_win=true, decrement
   if is_win=false. Learning rate = 0.05.

10. **Pruning** — If utility_score < 0.3, set `is_active = false`. Unit
    excluded from MMR search space. Can be reactivated if score recovers.

**Phase 4: SOUL.md Evolution**

11. **Immutable/mutable markers** — Add `<!-- MUTABLE -->` before Section XIII
    (Action Triggers). Everything else immutable by default. GEPA parser only
    touches mutable sections.

12. **Teacher LLM** — `teacher_critique()` in feedback.rs. Takes episode trace
    (reasoning, market context, outcome). Returns single-sentence heuristic.
    Same mimo v2.5 pro model ("model empathy"). Max reasoning temperature.

13. **GEPA mutation loop** — Parse SOUL.md mutable sections. Teacher LLM
    generates textual gradient. Apply mutation → SOUL_v_Next.md. Sandbox
    validates against train set (40 scenarios). Pareto gatekeeper checks
    val set (20 scenarios). Accept if Pareto optimal. Reject + store failed
    gradient if not.

14. **soul_versions table** — Schema: version_id (hash), timestamp, content,
    pareto_scores (JSON), parent_version_id, status (active/rejected/rolledback).
    Every mutation creates a row. Active version is loaded into prompt.

15. **Auto-rollback** — If CUSUM NegativeShift detected post-deployment,
    query soul_versions for last stable version. Auto-revert. Log rollback
    reason.

**Phase 5: Infrastructure**

16. **SQLite backup** — Add backup function to engine.rs startup + every 100
    trades. `sqlite3 data/memory.db ".backup data/backups/memory_{timestamp}.db"`.
    Keep last 7 backups. Rotate oldest. Also backup test_memory.db.

17. **WS exponential backoff** — Modify websocket.rs `connect()`. Current
    fixed 5s → exponential: 1s, 2s, 4s, 8s, 16s, 30s (cap). Reset on
    successful connection. Add jitter (±20%) to prevent thundering herd.

18. **Scenario expansion signature** — Build `failure_to_scenario()` in
    scenarios.rs. Takes episode data, returns Scenario struct. Defer
    implementation to after live trading produces real failures. Function
    signature + stub only.

**Phase 6: Validation + Integration**

19. **Train/val split** — Hard-code in scenarios.rs. First 40 scenarios = train
    set. Last 20 = val set. Ensure every category represented in both.

20. **Targeted remediation** — After baseline run, identify worst category.
    Next 5 GEPA iterations only evaluate that category. Re-integrate after
    improvement.

21. **Live correlation tracking** — Monthly Spearman rank correlation between
    sandbox category scores and live P&L per category. If divergence detected,
    adjust GARCH parameters.

22. **savant train --cycle full** — CLI command that runs all 6 phases in
    sequence. Scheduled nightly or on CUSUM alert.
