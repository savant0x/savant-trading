## Summary

Kraken has a token-based rate limit system. Public endpoints cost 1-2 tokens each. Without rate limiting, the bot could receive HTTP 429 responses and get temporarily blocked. Currently no rate limiting exists.

## Detailed Description

### Problem

`KrakenClient` makes HTTP requests without tracking rate limit consumption. On 5m intervals this is fine (~1 request per 5 minutes), but adding WebSocket, multi-timeframe fetching, or faster timeframes could hit limits.

### Expected Behavior

- Track rate limit tokens consumed per request
- Respect `Retry-After` header on 429 responses
- Configurable request delay between API calls

### Evidence

```
data/kraken.rs — no rate limit tracking, no 429 handling
```

## Impact Assessment

### Affected Components

- `src/data/kraken.rs` — add rate limit tracking, 429 retry logic

### Risk Level

- [x] Low: Not a problem at current 5m polling rate. Will matter with WebSocket or faster timeframes.

## Proposed Solution

### Steps

1. **Rate limit tracking** — Add `last_request_time: Instant` to `KrakenClient`. Enforce minimum 1s between requests.
2. **429 handling** — On HTTP 429, read `Retry-After` header, sleep, retry.
3. **Verify** — build + clippy + fmt

### Verification

- Verify requests are spaced by at least 1s
- Verify 429 responses trigger retry
