## Detailed Description

### Problem

1. **4 stub modules** violate ECHO Protocol — `funding_rates.rs`, `liquidation.rs`, `flows.rs`, `news.rs` are all empty TODO stubs
2. **No news ingestion** — the agent has no visibility into breaking news, regulatory announcements, or social sentiment
3. **No end-to-end test** — the AI pipeline has never been validated from data → context → LLM → decision
4. **Knowledge embedded in binary** — updating knowledge requires recompilation
5. **No trade history in AI context** — the agent can't learn from its own past decisions
6. **ECHO Protocol not in system prompt** — the AI doesn't know the project's operating principles
7. **No structured monitoring** — can't see the full decision chain in logs
8. **Single-pair insight** — insight only refreshes for the first configured pair

### Expected Behavior

- All 4 stub modules replaced with real API integrations
- RSS news feeds polled every 5 minutes, parsed, and injected into AI context
- `--dry-run` CLI flag makes one AI call and prints the full pipeline output
- Knowledge loaded from external `knowledge/` directory, not compiled into binary
- Last 10 trades included in AI context for self-learning
- ECHO Protocol trading rules embedded in system prompt
- Structured log output showing: conditions → knowledge selection → prompt summary → decision
- Insight refreshes for ALL configured pairs, not just the first

### Root Cause

Stub modules were scaffolded but never implemented. No RSS integration was considered. The AI pipeline was built incrementally without end-to-end validation.

## Proposed Solution

### Phase 1: Stub Elimination (4 files)

1. **`src/insight/funding_rates.rs`** — CoinGlass API integration
   - If `COINGLASS_API_KEY` env var set → make real HTTP call, parse response
   - If no key → log warning, return empty `FundingData`
   - Endpoint: `https://open-api-v3.coinglass.com/api/futures/funding-rate/current`
   - Parse: funding_rate, open_interest, long_short_ratio
   - Error handling: retry 2x, exponential backoff, log all failures

2. **`src/insight/liquidation.rs`** — CoinGlass liquidation API
   - Same pattern as funding_rates
   - Endpoint: `https://open-api-v3.coinglass.com/api/futures/liquidation`
   - Parse: long/short liquidation amounts, cluster prices

3. **`src/insight/flows.rs`** — blockchain.info (free) + CryptoQuant (key optional)
   - Always fetch from blockchain.info (free, no key):
     - `https://blockchain.info/q/getblockcount` → block height
     - `https://blockchain.info/q/unconfirmedcount` → mempool size
     - `https://blockchain.info/q/24hrtransactioncount` → tx count
   - If `CRYPTOQUANT_API_KEY` set → also fetch exchange flows
   - Return combined FlowData

4. **`src/insight/news.rs`** — RSS feeds + CoinGecko trending + CryptoPanic
   - Always fetch CoinGecko trending (free, no key)
   - Always fetch RSS feeds (free, no key) — see RSS section below
   - If `CRYPTOPANIC_API_KEY` set → also fetch CryptoPanic news
   - Derive sentiment score from trending coin price changes

### Phase 2: RSS Pipeline (2 files)

5. **`Cargo.toml`** — Validate `quick-xml` addition (already done)
6. **`src/insight/rss.rs`** — New module:
   ```rust
   pub struct RssFeed {
       pub url: String,
       pub name: String,
       pub category: RssCategory,
   }
   pub enum RssCategory { Macro, Defi, Institutional, Bitcoin }
   pub struct RssItem {
       pub title: String,
       pub link: String,
       pub pub_date: Option<DateTime<Utc>>,
       pub description: String,
       pub categories: Vec<String>,
       pub source: String,
   }
   pub async fn fetch_all_feeds(client: &reqwest::Client) -> Vec<RssItem> { ... }
   pub fn filter_relevant(items: &[RssItem], pairs: &[String]) -> Vec<&RssItem> { ... }
   ```
   - Poll all 8 feeds concurrently with `tokio::join!`
   - Randomized jitter (±30s) on polling interval
   - Deduplicate by URL
   - Score relevance by keyword matching against trading pairs
   - Parse with `quick-xml` — handle malformed XML gracefully

7. **`src/insight/mod.rs`** — Add `pub mod rss;`
8. **`src/insight/aggregator.rs`** — Add RSS to refresh cycle

### Phase 3: System Improvements (5 files)

9. **`src/main.rs`** — Add `--dry-run` CLI flag
   - Makes ONE AI call with current market data
   - Prints full pipeline: conditions → knowledge selection → prompt → LLM response → parsed decision
   - Exits after one cycle
   - Useful for testing and debugging

10. **`src/agent/knowledge/loader.rs`** — External knowledge loading
    - Load from `knowledge/` directory at project root (configurable)
    - Falls back to embedded (compiled-in) if directory not found
    - Same JSON format, same KnowledgeBase struct
    - Allows iterating on knowledge without recompiling

11. **`src/agent/context_builder.rs`** — Trade history in context
    - Add `recent_trades: &[TradeRecord]` parameter to `build_context()`
    - Format last 10 trades as: "Trade #1: BTC/USD LONG @ 65000 → 66500, PnL: +$150 (2.3%), 2026-05-30"
    - Include win rate, avg R:R, total PnL from recent trades
    - AI can use this to calibrate confidence and adjust behavior

12. **`src/agent/prompts.rs`** — ECHO Protocol in system prompt
    - Add trading rules from ECHO.md to the system prompt
    - Key rules: sell into strength (80% at 10-20% gain), 3 losses = stop, don't marry positions
    - Four-factor performance model (setup, process, market, trader)
    - Self-leadership principles

13. **`src/engine.rs`** — Multi-pair insight refresh
    - Change insight refresh from `pairs[0]` to loop over ALL pairs
    - Each pair gets its own insight data
    - Agent evaluates each pair independently
