## Summary

The action test runs 60 scenarios through the LLM and dumps output to console.
Nothing is persisted. The agent starts from zero every run. The episodic memory
system (EpisodicMemory, Brier score, CUSUM, experience replay) is fully built
and wired in the live engine but completely absent from the test harness. Every
test run should train the agent so it accumulates experience, calibrates
confidence, tracks edge per category, generates lessons from failures, and
progressively increases difficulty as it improves. This is the foundation of
a 24/7/365 autonomous agent — it must learn from every decision.

## Environment

- **OS:** Windows (win32)
- **Language/Runtime:** Rust 1.91
- **Tool Versions:** cargo 1.91, sqlx 0.8 (SQLite WAL)
- **Commit/State:** v0.4.2, knowledge selection + streaming + context_tags

## Detailed Description

### Problem

1. **No persistence** — Action test fires scenarios, prints results, exits.
   Every run starts from zero. No learning, no memory, no calibration.

2. **Memory system unused in tests** — EpisodicMemory (SQLite WAL),
   calibration.rs (Brier score), cusum.rs (edge decay), replay.rs (experience
   replay) are all built and wired in the live engine. None used in tests.

3. **No training loop** — The agent has persistent memory but it's never
   trained. Running 100 scenarios should produce 100 episodes. Running again
   should produce 200 total, with the agent seeing its own track record.

4. **No calibration** — Brier score measures confidence vs accuracy. Without
   capturing (confidence, is_win) pairs, the agent can never calibrate.

5. **No progressive difficulty** — All scenarios run regardless of agent skill.
   The agent should start with Easy scenarios and progress to Extreme as its
   Brier score improves.

6. **No auto-lesson generation** — High-conviction failures should generate
   experience replay lessons automatically. The agent should learn from its
   mistakes without manual intervention.

7. **No convergence detection** — No way to know when the agent has learned
   enough. Training should stop when Brier score plateaus.

8. **No confidence distribution** — No visibility into whether confidence
   levels are well-calibrated. "You said 80% on 10 trades, right on 6" is
   never shown.

9. **No CUSUM per category** — Edge decay tracking is per-pair only. The agent
   should know "I'm good at Trend Bull, bad at Crash scenarios."

10. **Only 60 scenarios** — Need 100 for meaningful training data.

11. **Compilation bugs** — `run_action_test` has format string errors and
    missing functions (expected_is_trade).

### Expected Behavior

Every test decision → captured as episodic memory episode. Next run → agent
sees its own track record via 6th prompt layer. Over runs → confidence
calibrates, edge is tracked per category, difficulty increases, lessons are
auto-generated from failures. Training stops when Brier converges.

### Root Cause

The action test was built as a fire-and-forget evaluator. The memory system
was built for the live engine. Nobody wired them together or built the
training infrastructure around them.

## Perfection Loop

### RED — All Issues

1. **Shared database contamination** — If test writes to `memory.db`, test
   episodes pollute live trade history. Brier, win rates, CUSUM contaminated.

2. **is_win semantics** — Live engine sets `is_win` after trade closes (P&L
   known). Tests need proxy: did agent's action match expected_action?
   Expected actions: "Buy (High Conviction)", "Hold / Take Profit", etc.

3. **Status field mixing** — Test episodes with status "executed"/"held" would
   be indistinguishable from live episodes.

4. **Memory context threshold** — `format_memory_prompt()` returns empty if
   `total_trades < 5`. First 4 runs get no memory context.

5. **No progressive difficulty logic** — All scenarios run regardless of skill.

6. **No auto-lesson generation** — High-conviction failures are not captured
   as lessons for future runs.

7. **No convergence detection** — No stopping criteria for training.

8. **No confidence distribution** — No calibration visibility.

9. **No CUSUM per category** — Edge tracked per-pair only.

10. **Only 60 scenarios** — Need 100.

11. **Compilation bugs** — Format strings, missing expected_is_trade().

### GREEN — Solutions

1. **Separate DB** — `sqlite:data/test_memory.db`. Full isolation.

2. **expected_is_trade()** — `fn(expected_action: &str) -> bool`.
   Lowercase, contains "buy"/"sell"/"short"/"add" → true.
   Contains "hold"/"no trade"/"reduce"/"take profit" → false.

3. **Status prefix** — `"test_action"` / `"test_hold"`. Enables
   `WHERE status LIKE 'test_%'` filtering.

4. **Acceptable** — First run trains, second run learns. Log when inactive.

5. **Progressive difficulty** — After each run:
   - Brier < 0.15 → unlock Extreme
   - Brier < 0.25 → unlock Hard
   - Brier < 0.35 → unlock Medium
   - Brier >= 0.35 → Easy only
   Filter scenarios by unlocked difficulty before each run.

6. **Auto-lesson generation** — After each run, query episodes where
   `is_win = false AND confidence > 0.7`. For each, insert into
   `experience_replay_lessons` with error_type "high_conviction_failure"
   and heuristic summarizing the mismatch.

7. **Convergence detection** — Track Brier score across runs. Stop if
   improvement < 0.02 for 3 consecutive runs. Print "CONVERGED" message.

8. **Confidence distribution** — After each run, bucket predictions by
   confidence (0-25%, 25-50%, 50-75%, 75-100%). For each bucket: count,
   accuracy, avg confidence. Print calibration table.

9. **CUSUM per category** — Store scenario category in episode's
   `condition_tags[0]`. After each run, calculate win rate per category.
   Print category edge table.

10. **40 more scenarios** — Generate by varying mock data on existing
    scenario structures: different fear/greed, funding, on-chain combos
    across Easy/Medium/Hard/Extreme difficulty levels.

11. **Fix compilation** — Format strings, write expected_is_trade(), verify
    build + test + clippy.

### AUDIT — Five Questions

1. ALL cases? ✅ Separate DB. New runs start empty, accumulate.
2. 1000 agents? ✅ Each gets own test_memory.db. Stateless functions.
3. Hostile attacker? ✅ Separate DB, no user input in parsing.
4. 2 years? ✅ Clean boundary. Status prefix enables filtering. Schema unchanged.
5. Industry standard? ✅ Train/test split, Brier calibration, progressive
   difficulty, auto-lesson generation — all standard ML/RL practice.

## Lessons Learned

Persistent memory without a training loop is an engine without fuel. The
infrastructure is worthless if nothing feeds it. Every decision the agent
makes — live or test — must be captured so it accumulates experience.
Progressive difficulty ensures the agent is always challenged at its current
skill level. Auto-lesson generation ensures it learns from failures without
manual intervention. Convergence detection ensures training stops when
there's nothing left to learn.
