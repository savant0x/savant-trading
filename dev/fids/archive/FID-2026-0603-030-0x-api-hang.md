## Summary

The 0x API intermittently hangs during swap execution. The AI makes a valid Buy decision, position sizer approves, but the `build_swap_tx()` call to 0x never returns. The 60s engine timeout catches it, but the swap never executes. This has blocked every swap attempt since the engine went live.

## Environment

- **OS:** Windows 11
- **Language/Runtime:** Rust 1.94, tokio
- **Tool Versions:** reqwest 0.12, 0x API v2 permit2
- **Chain:** Arbitrum (chain_id 42161)
- **Commit:** `fc03292` (main)
- **Wallet:** `0x543ca0434b84ad38c858d2d178d2082521711fbc`

## Detailed Description

### Problem

Every swap attempt hangs at the 0x API call:

```text
[Savant Trading] [06-03-2026 3:05 PM] [0x API] Calling: 0xaf88d065... -> ADA amount=9541800
(nothing after this — 60s timeout fires)
```

The `[0x API] Quote OK:...` line never prints. The `build_swap_tx()` call hangs indefinitely.

### Expected Behavior

`build_swap_tx()` should return within 10-15 seconds with a valid swap quote. If the API is down or returns an error, it should fail fast (not hang).

### Root Cause

The `reqwest::Client` has a 30s timeout, but it doesn't catch all hang scenarios:

1. **DNS resolution hang** — `reqwest` timeout applies to the request, not DNS
2. **TLS handshake hang** — Can hang indefinitely if the server accepts TCP but never responds to TLS
3. **Connection pool exhaustion** — If all connections are in use, new requests queue indefinitely
4. **0x API rate limiting** — May accept the connection but never send a response

The 60s `tokio::time::timeout` in `place_order()` catches the hang, but by then the opportunity is lost.

## Proposed Solution

### Approach

Add a `tokio::time::timeout` wrapper directly around the `build_swap_tx()` call in `execute_swap()`. This provides a fast-fail at the API level (15s) before the engine-level timeout (60s). Also add retry logic that retries on timeout (not just on gas/nonce errors).

### Steps

1. Wrap `self.backend.build_swap_tx(&swap_params)` in `tokio::time::timeout(15s, ...)`
2. If timeout, log and retry (up to 3 attempts)
3. If all retries fail, return error to engine
4. Build and test

## Lessons Learned

- `reqwest` timeout doesn't cover all hang scenarios (DNS, TLS, connection pool)
- Always add `tokio::time::timeout` around external API calls, not just at the engine level
- Fast-fail at the API level (15s) is better than slow-fail at the engine level (60s)
- The 0x API on Arbitrum is intermittently unreliable — need fallback to 1inch
