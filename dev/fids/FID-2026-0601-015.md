## Summary

Two independent Gemini Deep Research reports analyzed the Savant trading agent's configuration against MiMo v2.5 Pro's architecture (1M context, 128K output, MoE with 384 experts, chain-of-thought reasoning). Both reports identified critical flaws in risk management, fee assumptions, temperature, prompt architecture, token budgets, and training methodology. This FID consolidates all recommendations into a single implementation plan. The $50 starting budget requires full deployment (20%/position, 5 positions) to overcome fee drag — safety comes from tighter daily loss/drawdown limits and stop losses, not from shrinking positions.

---

## Detailed Description

### Issue 1: Fee rate is wrong (CRITICAL)

**Evidence:** Config says `fee_rate = 0.0026` (0.26% Kraken taker). Actual Kraken Pro base tier is **0.40% taker / 0.25% maker**. At $50 balance, we're in the lowest volume tier.

**Impact:** Every R:R calculation is overstated. A trade showing 1.5:1 R:R at 0.26% fees is actually ~1.1:1 at 0.40%. The agent takes trades that are mathematically negative EV.

**Fix:** Update `fee_rate = 0.0040`. Add maker order support (limit entries at 0.25%) with `order_type` field in JSON output. Save 37.5% on fees by using limits.

### Issue 2: Risk parameters too loose for small account (CRITICAL)

**Evidence:** `max_daily_loss = 0.20` ($10), `max_drawdown = 0.40` ($20). At $50, a 20% daily loss is catastrophic. A 40% drawdown needs 66% gain to recover.

**Fix:** Tighten safety nets while keeping full deployment:
- `max_daily_loss = 0.10` (10% = $5 at $50)
- `max_drawdown = 0.20` (20% = $10 at $50)
- `min_rr_ratio = 2.0` (was 1.5 — higher R:R compensates for full deployment risk)
- Keep `max_risk_per_trade = 0.20` and `max_positions = 5` — must be fully deployed

### Issue 3: Temperature too high for structured JSON output (HIGH)

**Evidence:** Both reports agree. Current `temperature = 0.7` introduces too much entropy for JSON generation. Reasoning models need some stochasticity for chain-of-thought, but 0.7 causes hallucinated prices and logic drift.

**Fix:** `temperature = 0.6`, add `top_p = 0.95`. The 0.6 temperature is the mathematical optimum for MoE reasoning models emitting structured output.

### Issue 4: Token budgets underutilize 1M context (HIGH)

**Evidence:** `knowledge_token_budget = 8000` chars (~2K tokens). `context_window_candles = 100` (8.3 hours). MiMo has 1M context window — we're using <1% of it.

**Fix:**
- `knowledge_token_budget = 20000` chars (~5K tokens)
- `context_window_candles = 500` (2 days of 5m data)
- Add 1D candles for macro context

### Issue 5: Timeout too short for reasoning model (HIGH)

**Evidence:** `timeout_secs = 180`. MiMo's chain-of-thought reasoning can take 200-300s for complex market analysis. The 180s timeout truncates reasoning, producing incomplete JSON.

**Fix:** `timeout_secs = 300`

### Issue 6: Prompt architecture incompatible with reasoning models (HIGH)

**Evidence:** With-memory report: "Reasoning models suffer severe performance degradation when subjected to system prompts or rigid few-shot structures. It disrupts their native reinforcement learning alignment." Current 6-layer prompt uses a system prompt.

**Fix:** Flatten into single user prompt with XML-style tags for structure:
```xml
<identity>You are Savant...</identity>
<risk_constraints>...</risk_constraints>
<strategy_knowledge>...</strategy_knowledge>
<market_data>...</market_data>
<memory_context>...</memory_context>
<output_format>...</output_format>
```

### Issue 7: Structured reasoning phase (HIGH)

**Evidence:** Both reports recommend forcing the model to organize its thinking into specific analytical sub-routines. Currently the model reasons freely, which can scatter logic across conflicting signals.

**Fix:** Add to prompt:
```
Think through these steps in order:
1. Multi-Timeframe Alignment — does the macro trend support this trade?
2. Order Book Evaluation — is there institutional flow confirming?
3. SOUL.md Trigger Verification — count aligned triggers per direction
4. Conflict Resolution — if triggers conflict, HOLD
5. Risk Calculation — Kelly fraction, position size, R:R
```

### Issue 8: Confidence calibration via Isotonic Regression (HIGH)

**Evidence:** 0-25% confidence bucket has 18% accuracy. The model outputs trades with near-zero confidence — token generation noise overrides analytical intent.

**Fix:**
1. Implement Isotonic Regression calibration layer using 1,134+ episodes
2. Map raw LLM confidence → calibrated probability
3. Hard floor at 50% calibrated confidence (was 40% raw)
4. Update after every training batch

### Issue 9: Maker order support (MEDIUM)

**Evidence:** Currently all orders are market (taker at 0.40%). Limit orders (maker at 0.25%) save 37.5% on fees. At $50, this compounds significantly over hundreds of trades.

**Fix:**
1. Add `order_type: "LIMIT" | "MARKET"` to JSON output schema
2. Default to LIMIT unless crisis-level anomaly
3. Update paper trader to handle limit orders
4. Update prompt to instruct agent on limit entry strategy

### Issue 10: Multi-timeframe expansion (MEDIUM)

**Evidence:** Current timeframes: 5m, 1h, 4h. Missing daily/weekly for macro context. Trading a 5m bullish setup against a daily bearish trend significantly degrades win probability.

**Fix:**
- Add `1d` to timeframes array
- Send daily candles in context for macro regime detection
- Instruct model: "Daily timeframe is a binary directional filter"

### Issue 11: Granular session awareness (MEDIUM)

**Evidence:** Current sessions are generic names (Asian/European/US). Both reports recommend UTC-based liquidity mapping:

| UTC Window | Liquidity | Strategy |
|------------|-----------|----------|
| 13:00-17:00 | Peak (US-EU overlap) | Momentum, breakouts |
| 08:00-13:00 | High (EU morning) | Trend continuation |
| 17:00-22:00 | Moderate (US post-overlap) | Mean reversion |
| 02:00-06:00 | Trough (deep Asian) | Reduce size, widen stops |

**Fix:** Update session module to use UTC liquidity profiles. Add confidence penalty (+15%) for breakout trades during 02:00-06:00 UTC.

### Issue 12: Garman-Klass volatility estimator (MEDIUM)

**Evidence:** Current ATR uses only close prices. Garman-Klass uses OHLC, providing more accurate volatility measurement.

**Fix:** Add Garman-Klass calculation to indicator engine. Use for dynamic stop-loss widths that expand during volatile regimes and compress during ranging.

### Issue 13: Four-factor causal attribution for losses (MEDIUM)

**Evidence:** Current training records win/loss but not *why*. The without-memory report recommends classifying losses: Setup, Process, Market, Trader.

**Fix:** After each loss in training, prompt the agent to classify:
- **Setup** — Wrong setup selection
- **Process** — Execution error (wrong size, wrong entry)
- **Market** — Market not in our favor
- **Trader** — Emotional/revenge trading

Inject causal attribution into memory context for next 5 episodes.

### Issue 14: Historical tick data training (MEDIUM)

**Evidence:** Synthetic scenarios plateaued at Brier 0.24. The without-memory report says training on real historical data (2021 blow-off, 2022 capitulation, 2023 compression) is the only way to cure short bias and Trend Bull weakness.

**Fix:**
1. Fetch historical OHLCV from Kraken API (free, no key)
2. Cache 1 year of 5m data (~105K candles)
3. Replay through training pipeline with real market conditions
4. Keep synthetic scenarios for edge cases (flash crashes, depegs)

### Issue 15: JSON schema constrained generation (MEDIUM)

**Evidence:** 5% streaming parse errors. The with-memory report recommends using `response_format: { "type": "json_schema" }` if OpenGateway supports it. Forces valid JSON at the inference level.

**Fix:**
1. Test if OpenGateway passes through `response_format` parameter
2. If supported, enforce TradeDecision JSON schema
3. If not supported, implement incremental JSON repair (AST-level)

### Issue 16: response_format support (LOW — conditional)

**Evidence:** MiMo v2.5 Pro supports structured output natively. If OpenGateway proxies this through, we can eliminate parse errors entirely.

**Fix:** Test with a simple request. If supported, switch to structured output. If not, implement json_repair crate for streaming AST repair.

### Issue 17: Remove system prompt (LOW — needs testing)

**Evidence:** With-memory report says reasoning models degrade with system prompts. But our system prompt is critical for trading rules.

**Fix:** A/B test: run 20 scenarios with system prompt, 20 without (user-only with XML tags). Compare Brier scores. If user-only is better, switch permanently.

## Lessons Learned

1. **Never trust fee assumptions** — 0.26% vs 0.40% sounds small but invalidates every R:R calculation.
2. **Small accounts need different risk framework** — Kelly/2% assumes you can survive losing streaks. At $50, you can't. Full deployment + tight safety nets is the correct approach.
3. **Reasoning models need different prompt architecture** — System prompts and few-shot examples degrade MoE reasoning models. XML-tagged user prompts work better.
4. **Token budgets must match model specs** — 8K chars on a 1M context model is like driving a Ferrari in first gear.
5. **Confidence calibration is a post-hoc math problem** — Don't trust the LLM's self-reported confidence. Calibrate it with Isotonic Regression using historical outcomes.
