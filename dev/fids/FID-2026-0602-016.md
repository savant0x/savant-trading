## Detailed Description

### Problem

The `ExecutionEngine` trait defines 4 methods: `place_order()`, `close_position()`, `open_positions()`, `balance()`. Only `PaperTrader` implements this. There is no real Kraken execution path. The engine cannot place actual orders.

### Expected Behavior

A `KrakenTrader` that:
1. Places real limit/market orders on Kraken via REST API
2. Tracks fills and updates positions
3. Manages stop-losses (standalone Kraken stop orders, not client-side)
4. Handles partial fills, order rejection, and API errors
5. Polls for order status updates
6. Syncs balance from Kraken account

### Root Cause

No Kraken private API client exists. The current `KrakenClient` only uses public endpoints (OHLC, order book, asset pairs). Private endpoints (AddOrder, CancelOrder, Balance, OpenOrders, ClosedOrders) require HMAC-SHA512Á≠æÂêç and are not implemented.

### Evidence

## Proposed Solution

### Approach

Build a `KrakenTrader` that implements `ExecutionEngine`. Use Kraken's REST API private endpoints with HMAC-SHA512Á≠æÂêç. Start with market/limit orders, add stop-loss management, then scale-out logic.

### Steps

1. **Kraken Private API Client** ‚Äî Add `sign()` method (HMAC-SHA512 + nonce). Add `add_order()`, `cancel_order()`, `get_balance()`, `get_open_orders()`, `get_trade_balance()`.

2. **KrakenTrader struct** ‚Äî Implement `ExecutionEngine`. Track positions locally. Sync with Kraken on startup.

3. **Order placement** ‚Äî Market orders for immediate fills. Limit orders for maker-fee routing (0.25% vs 0.40%). Stop-loss as standalone Kraken stop-market orders.

4. **Order status polling** ‚Äî Poll `OpenOrders` endpoint to detect fills. Update local position state. Handle partial fills.

5. **Balance sync** ‚Äî Fetch `Balance` endpoint on startup and after each trade. Reconcile with local state.

6. **Error handling** ‚Äî Retry on transient errors (rate limit, network). Halt on auth errors. Log all API calls.

7. **Engine integration** ‚Äî `config.mode.paper_trading = false` ‚Üí use `KrakenTrader`. Keep `PaperTrader` as fallback.

8. **Safety rails** ‚Äî Max order size cap ($10 at $50 balance). Daily loss halt. Position count limit. All enforced client-side BEFORE sending to Kraken.

9. **Dry-run test** ‚Äî Place a $1 limit order far from market (won't fill). Verify API auth works. Cancel immediately.

10. **Live test** ‚Äî Place a $5 market order on BTC/USD. Verify fill. Place stop-loss. Verify stop triggers.

## Lessons Learned

1. **Paper trading results are not live results.** $12K simulated return assumes perfect fills. Real execution has slippage, partial fills, and latency.
2. **Exchange-side stops are mandatory.** Client-side stops die with the process. Kraken stop-market orders survive crashes.
3. **HMACÁ≠æÂêç is the hard part.** The nonce must be strictly increasing. Clock drift can cause auth failures. Need monotonic counter, not timestamp.
4. **Rate limits are per-endpoint-type.** Orders and balance have separate 15s decay windows. Can't just use a single rate limiter.
5. **Minimum order sizes vary per pair.** BTC/USD might be 0.0001 BTC. DOGE/USD might be 10 DOGE. Must fetch from AssetPairs API.
