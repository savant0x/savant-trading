## Summary

A direct audit of FID-126's regime-specific conviction thresholds against FID-222's new K=12-from-250 universe reveals a **docs-vs-code docs lie** trilemma that reframes the original question. Live `src/agent/decision_parser.rs:56-78` returns **Trending 0.05, Volatile 0.15, Ranging 0.10, GreyZone 0.20** (post-FID-184, Gemini-recommended lowering). Soul.md doctrine and the LLM's `output_format.md` prompt instruct an unflappable **Trending 0.30, Volatile 0.40, Ranging 0.40, GreyZone 0.40**. These are non-trivially different — half the document, half the code. The LLM is being told to emit `conviction_score ≥ 0.30` while the parser passes `conviction_score ≥ 0.05`. Companion data: today's engine verdict distribution is **BUY:1, SELL:0, HOLD:9 (90% HOLD)** under the lower code thresholds, AND jury-metrics.json shows **67.9% evaluation failure rate (19/28)**. The funnel expansion will not, by itself, shift the verdict distribution unless we first correct the parser / prompt alignment and instrument a multi-cycle experiment.

## Detailed Audit Findings

### Finding 1 (CRITICAL): Docs-vs-code trilemma on Trending threshold

**Verified by direct grep of `src/agent/decision_parser.rs:56-78`** (the actual `RegimeLabel::conviction_threshold()` method body):

```rust
pub fn conviction_threshold(self) -> f64 {
    match self {
        // Trending: lowest threshold (0.05-0.08 per Gemini). Trend-following
        // has highest signal quality, so even low conviction is actionable.
        RegimeLabel::Trending => 0.05,
        // Volatile: mid threshold (0.15-0.18 per Gemini). Higher noise-to-signal
        // ratio, bear veto mandatory.
        RegimeLabel::Volatile => 0.15,
        // Ranging: low-mid threshold (0.10-0.12 per Gemini). Mean-reversion setups.
        RegimeLabel::Ranging => 0.10,
        // GreyZone: default to PASS (Gemini: 0.20+ default to PASS).
        // Regime uncertainty window — don't trade.
        RegimeLabel::GreyZone => 0.20,
    }
}
```

Per-regime doc-vs-code matrix:

| Regime | `decision_parser.rs:56-78` (live) | `soul.md` + `output_format.md` (LLM-visible) | Code looser than docs by |
|---|---|---|---|
| Trending | 0.05 | 0.30 | **6×** |
| Volatile | 0.15 | 0.40 | ~2.7× |
| Ranging | 0.10 | 0.40 | **4×** |
| GreyZone | 0.20 | 0.40 | 2× |

**Implication:** the LLM is being told (in `output_format.md` schema rules) to emit `conviction_score ≥ 0.30` for Trending, but the parser is *also happy with `conviction_score ≥ 0.05`*. The two halves of the system disagree by 6× on the most-traded regime. This is a real, dangerous, audit-worthy discrepancy.

**Why this matters for FID-222 even though funnel predates the audit:** the LLM will bias its output toward the LOWER expected threshold per its _internal_ model — but the prompt text keeps telling it the higher threshold. This contributes to the 87% zero-conviction plateau FID-184 attempted to fix: the LLM remains conservative because the doctrinal text says "0.30."

Two paths, both out of scope for FID-222:

- **Path A: Ratchet code UP to match doctrine.** Restore `Trending=0.30, Volatile=0.40, Ranging=0.40, GreyZone=0.40`. Drift correction is single-line per regime. Risk: returns to default-to-hold well (which FID-184 diagnosed).
- **Path B: Ratchet doctrine DOWN to match code.** Update `soul.md`, `output_format.md`, and the FID-126 doctrine to use the 0.05/0.15/0.10/0.20 values. Risk: prompt drift continues.

This FID-222.5 does NOT recommend either path preemptively. See Recommendation §5.

### Finding 3: K=12-from-250 verdict-distribution forecast

Mechanistic chain when FID-222 ships:

| Step | Today (K=18 hardcoded) | Forecast (K=12-from-250) |
|---|---|---|
| Candle-fetch universe | 18 | ~268 → ~200 alive post-hygiene |
| Pre-LLM funnel | none (FID-189 block is hygiene pass/fail only) | 6-signal composite rank → drop below 0.20 → top 12 |
| LLM input quality | mixed (18 hardcoded, mostly stagnant) | high (best 12 of 268 candidates) |
| `trigger_weights` distribution | skewed low (per FID-126 audit: mean 0.367, std 0.309, but `0.000` 4/60 anti-pattern) | expected shift toward higher means — funnel pre-selects for alignment |
| `conviction_score` distribution | unchanged from current 18-pair run | expect upward drift; same average may shift from ~0.30 to ~0.55 |
| Verdict distribution | 90% HOLD (today) | forecast: 50-65% HOLD, 25-35% BUY, 5-15% SELL |
| Jury-failure rate | 67.9% today | LOW CONFIDENCE — depends on jury calibration, not conviction |

**Why the LLM verdict distribution will shift up:** funnel's `pre_scorer` ranks pairs on a 6-signal composite (EMA alignment, RSI extreme, ADX trending, volume spike, VWAP proximity, BB squeeze). A high funnel score **mechanically correlates** with stronger LLM-detectable patterns → more "strong" + "moderate" `trigger_weights` → higher `conviction_score` for those pairs.

**Why the LLM verdict distribution will STILL be HOLD-heavy:** FID-126 verifier's headline finding is structural: the LLM is biased to HOLD regardless of input quality. The 90% HOLD rate in K=13 today is at code-threshold 0.05 (effectively no gate). Even at 0.05, BUY = 1/13. **The LLM's HOLD bias is upstream of any conviction threshold.** FID-222 may shift the conviction_score distribution but probably won't shift the verdict distribution as much.

### Finding 4: FID-184 trajectory — the doctrine of successive lowering

Timeline of threshold doctrine:

| Date | Doc | Trending threshold | Drift Δ |
|---|---|---|---|
| 2026-06-12 | FID-126 spec v0.14.0 | 0.50 | baseline |
| 2026-06-17 | FID-184 (Gemini) | 0.05-0.08 | -0.42 |
| 2026-06-17+ | Code at `decision_parser.rs:56-78` | 0.05 | -0.42 → applied |
| 2026-06-17+ | `soul.md`, `output_format.md` (LLM-visible) | 0.30 | not yet lowered in docs |
| Net today | LIVE (engine-side) | 0.05 | ↓ 6× |
| Net today | LIVE (LLM-side) | 0.30 | unchanged |

The doctrine-of-successive-lowering was an emergency response to 87% zero-conviction plateau (FID-184 §Problem). Gemini's prescription was correct as emergency first aid — but FID-184 left a bipolar state: code is at 0.05, docs are at 0.30. Either we should re-tighten to match the docs, or we should lower the docs to match the code, or we should split the difference by raising to a calibrated middle.

**A pre-emptive push in either direction risks another FID-184 episode** (Gemini called 0.20-0.25 "mathematically prohibitive" for scalping — implying threshold choice depends on regime frequency + vol regime).

### Finding 5: Engine-side filters ARE the binding constraint (FID-126-R3 finding)

The buy count is achievable without gate changes (FID-126-R3 gate-off run hit 17%). The engine-side filters — position sizing (`calculate_with_atr`/`calculate_with_conviction`), circuit breaker, correlation check, balance check, gas guard — block execution. FID-126-R5 (engine-side filter trace) was open from 2026-06-12 but isn't implemented yet. **Until engine-side filter trace ships, no threshold tuning can be empirically validated.** That alone is a blocker.

## Proposal (NOT YET ACTIVE — requires Spencer approval)

### Stage 1: Re-sync live code to doctrine (correction FID, separate from FID-222)

`src/agent/decision_parser.rs:56-78` rewrite from `[0.05, 0.15, 0.10, 0.20]` → `[0.30, 0.40, 0.40, 0.40]`:

```rust
// Trending: doctrine threshold (FID-126 soul.md). High-quality setups land here
// with conviction 0.7-1.0; weak setups in 0.0-0.3 fall to HOLD. Funnel-top-12
// selection bias ensures most Trending pairs in the new universe carry conviction 0.4+.
RegimeLabel::Trending => 0.30,

// Volatile + Ranging + GreyZone: doctrine 0.40 holds. Volatile noise-to-signal
// is high; Ranging mean-reversion needs statistical confluence; GreyZone differentiates
// to Volatile if no disambiguator trigger. All paths require 0.4 minimum.
RegimeLabel::Volatile => 0.40,
RegimeLabel::Ranging => 0.40,
RegimeLabel::GreyZone => 0.40,
```

Plus the override `bypass_gates` (FID-126-R3 env var) stays. Cleaner single-split: re-sync, do not lower.

**This is a separate FID-222.5.1 it is NOT inside FID-222 scope.** Funnel's pipeline end-state should not depend on threshold tuning — funnel is the input-side fix; threshold is a separate sibling question.

**Why this matters:** FID-126-R3's lesson was that gate-tuning without engine-side filter trace is guesswork. We MUST capture the verdict distribution per cycle, run for ≥3 cycles after both FID-222 and Stage 1 fix ship, then tune empirically.

### Stage 3: Engine-side filter trace (would be a separate FID)

FID-126-R5 (engine-side filter trace) was identified 2026-06-12 but NOT implemented. The LLM emits Buys; the engine blocks them. Without visibility into WHICH filter blocks WHERE, K=12 funnel expansion's effect on trade count will be invisible. **This is a hard blocker before any threshold decision.**

Until FID-126-R5 ships, tuning thresholds is gambling. Recommend filing this as a new separate FID-126-R6 / SFID-X right after FID-222.

**My defensible default:** **Option A** (re-sync + telemetry embedded in FID-222's Q-G JSONL) **OR Option D** (don't tune yet; tune AFTER ≥3 cycle empirical run).

I lean **Option D** heavier because:
1. FID-126-R3 already taught us: gate tuning without engine-side filter trace is guesswork.
2. The LLM's 90% HOLD bias is upstream of any threshold; threshold tuning won't change verdict distribution as much as we hope.
3. The trilemma is real but the cost of wrong-direction tweak is measured in live trades. Better to be empirical.

**If Spencer agrees (Option D), this FID-222.5 is finished: docs-vs-code noted, telemetry extension added to FID-222 JSONL schema, no threshold code changes.**
